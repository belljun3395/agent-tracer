# ADR-0027 — 로컬 에이전트도 서버 에이전트와 같은 기준을 지킨다

## 상태

채택됨

## 결정

규칙 생성은 사용자의 기계에서 데몬이 Claude Agent SDK로 직접 실행한다. 서버가 아니라 로컬인
이유는 규칙이 워크스페이스를 봐야 하기 때문이다. 어떤 명령이 실제로 있는지, 어떤 파일이
실재하는지는 이벤트 원장에 없다.

실행하는 곳이 다르다고 기준이 달라지지 않는다. 로컬 에이전트도 서버 에이전트와 같은
실행 형태를 따른다.

**조사 → 검증 → (한 번) 수리 → 확정 또는 빈 결과.**

계약이 잠그는 것도 같다. 커널의 계약 픽스처가 도구 인자와 출력 스키마와 예산과 합격 기준을
소유하고 테스트가 그것을 읽는다.

## 근거

로컬 에이전트가 내는 것은 제안이 아니라 **집행되는 규칙**이다. 사용자가 승인한 규칙은 다음
세션에서 턴을 막는다. 그래서 잘못된 규칙의 비용이 잘못된 레시피 후보보다 크다. 레시피는 안 쓰면
그만이지만 규칙은 사용자의 작업을 멈춘다.

그런데 이쪽이 오히려 검증이 없었다. 도구가 돌려주는 근거에 식별자가 없어 모델이 무엇을
보고 그 규칙을 냈는지 확인할 방법이 없었다. 없는 명령을 규칙으로 내도 저장되고 집행됐다.
프롬프트로는 막을 수 없다. 이것이 근거 장부 대조 검증을 두는 이유 그대로다.

로컬이라는 사실이 기준을 낮추는 근거가 되지 않는다. 오히려 로컬은 사용자의 토큰을 쓰고,
사용자의 워크스페이스에서 돌고, 사용자의 턴을 막는다. 셋 다 서버 에이전트에는 없는 위험이다.

## 구현

**근거가 식별자를 싣는다.** 턴 요약은 turnId를, 이벤트 근거는 eventId를 함께 낸다. 인용할 수
없는 근거는 근거가 아니므로 식별자가 없는 항목은 버린다.

**제안이 인용한다.** 규칙 제안은 자기가 본 turnId와 eventId를 함께 낸다. 규칙은 사용자가
꺼낸 요구를 검증하는 것이므로 턴 인용이 최소 하나 있어야 한다. 워크스페이스 파일에는
식별자가 없으므로 파일만 보고 낸 규칙도 그 요구가 어디서 왔는지는 대야 한다.

**근거 장부는 도구 응답만 먹는다.** 실행 하나가 근거 장부 하나를 갖고, 도구가 모델에게 실제로
돌려준 식별자만 올라간다. 실재하는 식별자라도 이 실행에서 도구가 돌려주지 않았으면 근거가 아니다.

**검증에 걸리면 한 번 수리한다.** 오류를 모델에게 돌려주고 다시 받는다. 그래도 근거가 서지
않는 제안은 버린다. 버려도 이미 쓴 돈은 보고한다.

**실패와 취소를 가른다.** 사용자가 취소했거나 리스를 잃은 실행은 실패가 아니다. 그 작업은 더
이상 우리 것이 아니다. 시간 초과는 실패이며 그때까지 쓴 비용을 함께 적는다.

**실행기는 포트 뒤에 있다.** 그래야 어댑터가 가짜 모델로 테스트된다. 실행기를 직접 부르면
결과 해석도 오류 분류도 실행 기록도 테스트할 수 없다.

**실행 기록은 서버 에이전트와 같은 어휘로 남긴다.** 스텝의 모양이 같아야 로컬 실행과 서버 실행을
나란히 놓고 볼 수 있다.

**서브프로세스는 사용자 전역 설정을 싣지 않는다.** 이 실행의 문맥은 저장소 파일과 태스크
이벤트로 채워지므로 공격자가 영향을 줄 수 있다. 그런 문맥을 읽는 실행이 사용자의 전역 훅을
권한 우회 모드로 실행하게 두지 않는다. 도구는 읽기 전용만 남긴다.

## 결과

- 로컬 규칙 생성기에 새 도구나 새 예산을 더하려면 계약 픽스처를 먼저 고친다.
- 근거 없이 나온 규칙은 저장되지 않는다. 사용자의 턴을 막는 것은 근거가 선 규칙뿐이다.
- 인용한 식별자는 합격 기준을 위한 값이므로 저장하지 않는다. 사람이 규칙을 검토할 때
  누르는 것은 그 규칙을 낳은 발화이지 모델이 본 이벤트 목록이 아니다.
- 로컬에는 오케스트레이션 엔진이 없으므로 재시도와 취소는 데몬이 직접 갖는다.
