# ADR-0012 — 아키텍처 매니페스트가 규칙을 단독으로 소유하고 규칙은 파일 이름을 모른다

## 상태

채택됨

## 결정

아키텍처 규칙의 서술은 `architecture.manifest.mjs` 한 곳에만 있다.
의존 그래프 검사기와 린터가 이 선언을 읽어 규칙을 계산하고, 경로 별칭 설정이 이 선언에서
생성된다. 같은 경계를 두 곳에 손으로 적지 않는다.

**아키텍처 규칙은 구체적인 파일 이름을 인용하지 않는다.** 규칙은 디렉터리 역할과 파일 접미사만으로
표현된다.

규칙의 종류는 셋뿐이다.

**구조 규칙.** 계층 방향, 슬라이스 독립, `config`와 `support`의 무지, 배포 단위 격리.
**접미사 규칙.** 유스케이스는 유스케이스를 부르지 않는다. 조회 컨트롤러는 명령 유스케이스를
부르지 않는다. 워크플로는 활동 구현과 어댑터를 모른다. 인바운드는 투영 단계를 모른다.
**기술 봉인 규칙.** ORM은 어댑터와 설정과 마이그레이션 밖으로 새지 않는다. 프레임워크는 모델과
포트에 없다. 스키마 검증기는 스키마 파일에만 있다.

품질 허용치는 0에서 시작하고 올리지 않는다. 300줄을 넘는 소스 파일, 테스트 없는 유스케이스,
주석 규칙 위반의 허용치가 전부 0이다.

## 근거

규칙이 특정 파일 이름을 알아야 한다면 그 파일이 잘못된 자리에 있는 것이다.
"이 소비자 파일은 저 핸들러 폴더를 몰라야 한다"는 규칙은 소비자와 핸들러의 관계를 디렉터리가
표현하지 못했다는 자백이다. 파일을 옳은 자리로 옮기면 그 규칙은 일반 규칙에 흡수되어 사라진다.

그래서 "규칙이 파일 이름을 인용하는가"가 구조의 건강 지표가 된다. 인용하는 규칙이 하나라도
있으면 구조를 고칠 때다. 새 규칙을 추가하고 싶어지는 순간이 곧 구조를 고치라는 신호다.

같은 경계를 여러 도구에 손으로 적으면 반드시 어긋난다. 어긋남을 잡으려고 또 다른 검사 스크립트를
만들게 되고, 그 스크립트가 세 번째 서술이 된다. 선언을 하나로 두고 규칙을 계산하면 어긋날 자리가
없어진다.

허용치를 0이 아닌 값으로 시작하면 그 값이 목표가 된다. 처음부터 위반이 없으므로 0에서 시작할 수
있고, 0은 지키기 쉽다. 1은 2가 되고 2는 25가 된다.

## 결과

- 매니페스트는 계층과 그 의존 방향, 파일 역할 접미사, 배포 단위와 별칭, 기술 봉인 목록을
  선언한다.
- 의존 그래프 설정 파일은 매니페스트를 읽어 금지 규칙을 계산한다. 생성 산출물이 아니므로
  동기화할 파일이 없다.
- 린터 설정도 같은 매니페스트를 읽는다. 편집기 안에서 즉시 피드백을 주는 역할이고,
  의존 그래프 검사기와 규칙 문안을 나눠 갖지 않는다.
- 경로 별칭 설정만 정적 파일이라 계산이 안 된다. 매니페스트에서 생성해 커밋하고, 지속적 통합이
  재생성 후 변경이 없는지 확인한다.
- 슬라이스와 계층 이름은 정규식 역참조로 일반화된다. 슬라이스를 추가해도 규칙은 늘지 않는다.
- 300줄을 넘기는 파일이 나오면 크기 때문이 아니라 책임 때문에 나눈다. 단일 책임인데 긴 파일은
  책임이 하나임을 먼저 확인한다.
