# ADR-0019 — 웹은 여섯 레이어로 나눈다

## 상태

채택됨

## 결정

웹 대시보드는 여섯 레이어로 나눈다.

```
app/        조립, 라우팅, 프로바이더, 전역 스타일
pages/      라우트 단위 화면
widgets/    독립 UI 블록
features/   사용자 행동 단위
entities/   도메인 모델과 그 표현
shared/     UI 킷, 유틸, API 클라이언트, 설정
```

규칙은 둘이다.
상위 레이어만 하위 레이어를 import한다.
같은 레이어의 슬라이스끼리 직접 import하지 않는다.

운영 UI 문자열은 영어로 쓴다. 한국어 설명 문구는 단일 카탈로그가 소유하고, 화면은 그 카탈로그의
메시지를 렌더링한다. 화면 코드에 설명 문구를 직접 쓰지 않는다.

## 근거

프런트엔드에서 흔한 파탄은 화면 조각들이 서로를 부르기 시작하는 것이다. 목록 화면이 상세
화면의 내부를 알고, 상세 화면이 사이드바의 상태를 안다. 그러면 화면 하나를 고칠 때 어디까지
영향이 가는지 알 수 없다.

이 파탄을 규칙으로 막으려면 "이 화면은 저 화면만 조립할 수 있다" 같은 규칙을 화면 조합마다
하나씩 써야 한다. 화면이 늘면 규칙이 제곱으로 는다.

레이어를 세우면 이 규칙이 두 줄로 줄어든다. 위에서 아래로만 부르고 옆으로는 부르지 않는다.
목록 화면이 상세 화면을 부르고 싶으면 둘 다 위쪽 레이어가 조립한다.

레이어의 이름이 역할을 말하는 것도 중요하다. `widgets`는 그 자체로 완결된 UI 블록이고,
`features`는 사용자의 한 가지 행동이고, `entities`는 도메인 개념 하나의 표현이다.
새 코드를 어디에 둘지 이름이 답한다.

카피의 언어를 가르는 이유는 다르다. 운영 UI는 짧고 관례적인 영어가 자연스럽고, 설명 문구는
사용자의 모국어가 필요하다. 설명 문구를 화면에 흩어 놓으면 문체가 갈라지고 번역할 수 없다.
카탈로그 하나가 소유하면 문체가 하나가 되고 나중에 언어를 늘릴 수 있다.

## 결과

- 서버 상태를 가져오는 훅은 그 도메인의 `entities/` 안에 산다. 전역 상태 폴더에 모으지 않는다.
- HTTP 클라이언트와 실시간 연결은 `shared/`에 있다. 자원별 요청은 각 도메인이 갖는다.
- 순수 계산은 브라우저 API를 만지지 않는다. 효과는 바깥 경계에서 주입받는다.
- 화면에 인라인 스타일로 테마 값을 하드코딩하지 않는다. 유틸리티 클래스가 소유한다.
- 린터가 화면 코드의 한국어 리터럴과 카탈로그 우회를 잡는다.
