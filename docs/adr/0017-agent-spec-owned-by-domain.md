# ADR-0017 — 백엔드가 자기 명세를 갖고, 계약이 두 명세를 같게 만든다

## 상태

채택됨

## 결정

에이전트를 실행하는 백엔드는 둘이다. 언어 모델 SDK로 직접 실행하는 것과 파이썬 그래프로
실행하는 것이다. 각 백엔드가 자기 프롬프트와 자기 도구 명세와 자기 출력 모델을 갖는다.
파이썬은 워커가 보낸 프롬프트를 실행하는 것이 아니라 자기 프롬프트를 쓴다.

대신 두 명세가 같아야 하는 사실은 계약이 소유한다. 계약은 커널의 픽스처 파일이고,
두 언어의 테스트가 그 파일 하나를 읽는다. 한쪽 구현만 바꾸면 그쪽 테스트가 깨지고,
계약만 바꾸면 반대쪽이 깨진다. 초록으로 가는 길은 계약과 두 구현을 함께 고치는 것뿐이다.

계약이 강제하는 것은 넷이다.

- **도구 계약.** 도구 이름, 인자의 이름과 필수 여부, 기본값과 상하한, 열거값.
- **출력 스키마.** 필드 이름과 필수 여부와 길이 상한과 열거값. 도구 응답 본문도 같다.
- **예산.** 턴 한도, 토큰 한도, 비용 한도, 후보 개수 한도, 도구 호출 한도.
- **합격 기준.** 모델의 출력을 무엇으로 거부하는가. 거부하면 한 번 수리하고, 그래도
  근거가 서지 않으면 버린다.

계약이 강제하지 않는 것은 프롬프트의 문장과 실행 기계다. 그래프 위상, 도구 루프, 관측,
수리를 어떻게 다시 묻는지는 백엔드의 것이다. 다만 프롬프트가 말해야 하는 사실은 정해져
있다. 출력 필드의 계약, 예산, 금지 규칙, 근거 인용 규칙이다. 한쪽 모델만 모르는 규칙이
있으면 그것은 계약의 구멍이다.

## 근거

원래 이 문서는 도메인이 명세를 단독으로 소유하고 파이썬은 그것을 받아 실행만 한다고
정했다. 근거는 옳았다. 복사된 지식은 갈라지고, 갈라진 프롬프트는 백엔드마다 다른 결과를
낸다. 그러면 백엔드를 비교하는 실험이 무의미해진다.

실제로 그 일이 일어났다. 파이썬이 자기 프롬프트와 도구를 갖게 되자 도구 하나에 필터가
빠지고, 턴 예산이 달라지고, 인용 검증의 합격 기준이 갈라졌다. 예측한 그대로였다.

그러나 명세를 워커에서 실어 보내는 것이 답은 아니다. 프롬프트와 도구 스키마와 출력
스키마를 와이어로 보내면 파이썬의 검증 정책은 필드 이름을 잃고, 구조화 출력은 타입을
잃는다. 실행기가 잘하는 일을 못 하게 만들면서 지식의 동일성을 사는 셈이다.

지식이 갈라지는 것을 막는 데 필요한 것은 **한 벌의 소스**가 아니라 **한 벌의 계약**이다.
두 구현이 각자의 언어로 자기 명세를 쓰되, 무엇이 같아야 하는지를 기계가 검사하면 갈라짐은
커밋을 통과하지 못한다. 사람이 손으로 맞추는 중복은 갈라지지만, 테스트가 지키는 중복은
갈라지지 않는다.

이 선택의 대가는 정직하게 적는다. 프롬프트의 문장이 다르므로 두 백엔드의 비교는 순수한
실행 방식의 비교가 아니라 "그 백엔드와 그 백엔드의 프롬프트"를 묶어서 비교하는 것이다.
비교의 대상은 프롬프트 바이트가 아니라 최종 출력과 성공률과 근거 정확도와 비용과 지연이므로
이 대가를 감수한다.

## 결과

- 파이썬은 자기 프롬프트와 도구와 검증 정책을 갖고, 도구가 필요로 하는 데이터도 스스로 읽는다.
- 워커가 파이썬에 보내는 것은 실행 봉투와 결정적 사전 필터의 결과뿐이다. 모델과 데드라인과
  언어와 사용자 범위와 완료 창구다.
- 워커는 파이썬의 출력을 자기 스키마로 다시 검증한다. 백엔드를 믿지 않는다.
- 새 도구나 새 출력 필드나 새 예산을 더하려면 계약 픽스처를 먼저 고친다. 그러면 두 언어의
  테스트가 동시에 빨개지고, 둘 다 고쳐야 초록이 된다.
- 두 백엔드의 실행 궤적은 스텝 단위로 같은 형태여야 한다. 그래야 비교가 된다.
