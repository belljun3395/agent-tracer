# ADR-0011 — 팬아웃이 본질인 배포 단위는 기능 축으로 나눈다

## 상태

채택됨

## 결정

조회 모델 생성기의 최상위 축은 도메인이 아니라 기능이다.

```
project/    이벤트 원장에서 조회 모델을 생성한다
notify/     조회 모델 생성 결과를 알림으로 발행한다
index/      검색 인덱스를 채운다
export/     관측 파이프라인으로 내보낸다
```

기능 축이라도 계층 어휘는 같다. 각 기능이 다섯 계층을 그대로 갖는다.

`project/`의 진입점은 유스케이스 하나다. 배치를 받아 트랜잭션을 열고, 레코드마다 해당 조회 모델 생성
단계를 호출하고, 마지막으로 적용한 시퀀스를 갱신하고 커밋한다. 개별 생성 단계는 `*.projection.ts`이며
자기가 필요한 저장소 포트만 받는다.

## 근거

조회 모델 생성기가 하는 일은 이벤트 원장 배치 하나를 받아 여러 조회 모델을 한 트랜잭션 안에서
갱신하는 것이다. 타임라인과 실행과 레시피와 규칙의 조회 모델 생성은 함께 성공하거나 함께 실패한다.
트랜잭션 경계가 하나이므로
이들은 원래 한 몸이다.

여기에 도메인 축을 씌우면 넷이 형제 슬라이스가 되고, 형제는 서로를 부르지 못한다는 규칙에 걸린다.
그러면 이 금지를 우회하기 위해 조회 모델 생성 인터페이스와 레지스트리와 의존성 주입 장치를
만들게 된다.
그 장치는 아무 문제도 풀지 않는다. 우리가 스스로 만든 문제를 푼다.

축을 고르는 기준은 "무엇이 함께 변하는가"다. 조회 API에서는 태스크와 규칙이 따로 변하므로
도메인 축이 맞다. 조회 모델 생성기에서는 그것들이 함께 변하므로 기능 축이 맞다.

한 저장소 안에서 축이 다른 배포 단위가 섞이는 것은 일관성의 결함이 아니다.
축을 배포 단위마다 그 배포 단위의 변경 이유에 맞춘 결과다.

## 결과

- 조회 모델 생성 단계는 유스케이스가 아니다. 진입점이 될 수 없고 `inbound/`가 참조하지 못한다.
- 조회 모델 생성 단계 하나를 추가하는 것은 `project/application/projection/`에 파일 하나를 더하고 배치
  유스케이스가 그것을 호출하게 하는 일이다. 새 슬라이스를 만들지 않는다.
- 검색 인덱싱과 알림 발행과 관측 내보내기는 이벤트 원장 소비와 트랜잭션을 공유하지 않는다.
  그래서 따로 있는다. 각자 자기 소비자와 자기 포트를 갖는다.
- 회수기는 주기 실행이 인바운드다. 소비자가 아니다.
