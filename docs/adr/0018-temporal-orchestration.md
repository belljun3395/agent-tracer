# ADR-0018 — AI 작업은 내구성 있는 오케스트레이션 엔진이 실행한다

## 상태

채택됨

## 결정

AI 작업은 조회 API가 직접 실행하지 않는다. 조회 API는 작업을 접수하고 즉시 응답한다.
Temporal 워크플로가 실행 순서를 조정하고, 워커가 액티비티를 실행한다.

워크플로는 준비, 생성, 마무리의 세 단계를 조율한다. 재시도, 시간 초과, 취소는 엔진이 소유한다.
애플리케이션 코드가 재시도 루프를 갖지 않는다.

`*.workflow.ts`는 결정적이어야 한다. 액티비티 구현, 어댑터, 포트, 앱 설정, 시스템 API를
import하지 못한다.

## 근거

에이전트 호출은 수십 초에서 수 분이 걸리고 실패한다. 이것을 HTTP 요청 안에서 처리하면 요청이
타임아웃되고, 그 시점에 이미 지불한 토큰 비용은 회수되지 않는다.

직접 큐를 만들면 재시도, 중복 실행 방지, 취소, 진행 상황 조회를 전부 손으로 만들게 된다.
그 각각이 미묘하고, 미묘한 부분은 프로덕션에서만 드러난다.

내구성 있는 엔진은 이것들을 이미 갖고 있다. 워크플로가 죽어도 실행 기록에서 복구되고, 취소가
진행 중인 액티비티까지 전파된다.

재시도가 여러 층에 있으면 곱셈이 된다. 애플리케이션이 세 번, 엔진이 세 번 재시도하면 아홉 번
호출되고 아홉 번 과금된다. 재시도의 소유자를 엔진 하나로 못 박아야 한다.

워크플로의 결정성은 엔진의 외부 제약이다. 같은 실행 기록을 재생하면 같은 결정을 내려야 하므로,
워크플로 코드가 현재 시각이나 난수나 네트워크를 만지면 복구가 깨진다.

## 결과

- 언어 모델 SDK의 자체 재시도는 0으로 강제한다. 재시도는 엔진 단독 소유다.
- 작업의 상태 전이는 한곳이 소유한다. 여러 액티비티가 각자 작업의 상태를 바꾸지 않는다.
- 액티비티는 멱등해야 한다. 같은 액티비티가 두 번 실행되어도 같은 결과여야 한다.
  중복 과금을 막기 위해 공급자 호출에 멱등 키를 실어 보낸다.
- 취소는 진행 중인 도구 실행까지 전파된다. 취소의 원인은 원형 그대로 다시 던진다.
- 워크플로 번들의 진입 모듈을 명시한다. 결정성 위반은 규칙이 잡는다.
- 사용자는 작업의 진행 상황과 실행 기록과 비용을 화면에서 본다. 작업은 취소할 수 있다.
