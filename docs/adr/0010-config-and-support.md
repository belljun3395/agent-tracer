# ADR-0010 — 앱 전역 기술은 config가, 순수 유틸은 support가 소유한다

## 상태

채택됨

## 결정

배포 단위의 `src/` 아래에는 `domain/` 말고 두 폴더만 있다.

`config/`는 앱 전역의 기술 기반이다. 데이터베이스 연결, 메시지 브로커 배선, 인증 가드,
예외 필터, 응답 봉투, 헬스 서버, 로깅, 메트릭, 언어 모델 공급자 실행기가 여기 있다.
IO와 배선이 있고 도메인 어휘가 없다.

`support/`는 순수 유틸이다. 상태도 IO도 도메인 어휘도 없다. 입력을 받아 출력을 낸다.

`config/`와 `support/` 아래에는 유스케이스와 도메인 모델이 존재할 수 없다.
`config/`는 도메인을 import하지 못한다. `support/`는 `config/`와 도메인을 import하지 못한다.

## 근거

"공용"이라는 이름의 폴더는 반드시 오염된다. 이름이 아무것도 금지하지 않기 때문이다.
여러 도메인이 쓴다는 이유로 도메인 로직이 흘러들어오고, 그러면 그 폴더가 도메인의 변경에
끌려다니면서도 어느 도메인에도 속하지 않는 상태가 된다. 소유자가 없는 코드가 된다.

이름을 `config`와 `support`로 바꾸면 그 유혹이 사라진다. 규칙 판정 로직을 `config/`에 넣으려는
사람은 없다. 이름이 스스로를 지킨다.

셋 중 하나를 고르는 판단은 단순하다. 도메인 어휘가 있으면 `domain/`이다. 없고 IO나 배선이
있으면 `config/`다. 없고 순수하면 `support/`다.

## 결과

- 슬라이스의 `adapter/`와 앱의 `config/`는 이렇게 갈린다. 슬라이스가 소유한 포트를 구현하면
  `adapter/`이고, 도메인 포트 없이 기술 자체를 제공하면 `config/`다.
  데이터베이스 커넥션은 `config/`이고, 그 커넥션으로 태스크를 읽는 저장소는 태스크 슬라이스의
  `adapter/`다.
- 언어 모델 공급자 실행기는 `config/`에 있다. 프롬프트를 넣으면 구조화된 출력을 낸다.
  무슨 프롬프트인지는 모른다. 그것은 도메인이 안다.
- 여러 도메인이 함께 쓰는 무언가가 도메인 어휘를 담고 있다면, 그것은 `config/`도 `support/`도
  아니다. 각 도메인으로 복사한다.
