# ADR-0007 — 계층 어휘를 다섯으로 고정하고 기술은 파일 접미사가 말한다

## 상태

채택됨

## 결정

모든 배포 단위의 모든 슬라이스가 같은 다섯 계층을 쓴다.

`inbound/`는 구동 어댑터다. 바깥에서 들어오는 요청을 받아 응용 계층으로 넘긴다.
`application/`은 유스케이스다. 이 도메인이 하는 일이다.
`port/`는 아웃바운드 계약이다. 이 도메인이 바깥에 요구하는 것이다.
`adapter/`는 포트의 구현이다. 기술이 여기 있다.
`model/`은 순수 도메인이다. 프레임워크도 IO도 모른다.

기술의 차이는 디렉터리가 아니라 파일 접미사가 말한다. HTTP 컨트롤러, 메시지 소비자,
워크플로, 활동, 훅 진입점이 전부 같은 `inbound/` 자리에 온다.

의존 방향은 다음과 같다. `inbound`는 `application`과 `model`을 안다. `application`은 `port`와
`model`을 안다. `port`는 `model`만 안다. `adapter`는 `port`와 `model`과 기술을 안다.
`model`은 아무것도 모른다.

## 근거

역할이 같은데 이름이 다르면 규칙을 역할마다 다시 써야 한다. HTTP 앱의 컨트롤러와 소비자 앱의
컨슈머와 워커의 진입점은 전부 "바깥에서 들어오는 요청을 응용 계층으로 넘긴다"는 같은 역할이다.
이름이 같으면 규칙 하나가 셋을 덮는다.

이름이 다르면 규칙이 세 벌이 되고, 새 배포 단위가 생길 때마다 네 번째 규칙을 손으로 추가하게
된다. 규칙이 늘어나면 규칙을 읽지 않게 되고, 읽지 않는 규칙은 지켜지지 않는다.

기술은 파일 접미사가 말하므로 정보를 잃지 않는다. 오히려 한 슬라이스가 HTTP와 메시지를 동시에
받는 경우를 자연스럽게 표현한다.

## 결과

- `*.usecase.ts`는 응용 계층의 진입점이다. 공개 실행 메서드가 하나이고 다른 유스케이스를
  호출하지 않는다.
- `*.projection.ts`는 진입점이 아니라 유스케이스가 밟는 단계다. 유스케이스가 호출할 수 있고
  `inbound/`는 이를 직접 참조하지 못한다. 한 트랜잭션 안에서 여러 갈래로 갈라지는 처리에만 쓴다.
- 유스케이스가 다른 유스케이스를 부르고 싶어지면, 둘 중 하나는 유스케이스가 아니다.
  공통 부분을 `model/`로 내리거나 둘을 합친다.
- `application`이 `adapter`를 import하면 위반이다. 타입만 가져오는 경우도 마찬가지다.
  방향은 방향이다.
- 프레임워크 데코레이터는 `inbound/`와 `adapter/`에만 있다. `model/`과 `port/`는 순수 언어로
  쓴다.
