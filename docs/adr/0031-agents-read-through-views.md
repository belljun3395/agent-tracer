# ADR-0031 — 파이썬 에이전트는 읽기 전용 뷰로만 원장을 읽는다

## 상태

채택됨

## 결정

파이썬 에이전트 서비스가 자기 도구를 스스로 실행한다. 도구가 이름과 인자 스키마와 설명과
실행을 한 곳에서 소유한다.

도구가 필요로 하는 데이터는 **읽기 전용 뷰**에서 직접 읽는다. 파이썬은 기반 테이블을 읽지
않는다. 뷰가 두 배포 단위 사이의 계약이고, 뷰 정의는 마이그레이션이 소유한다.

실행 요청은 `userId`를 싣는다. 사용자 범위는 조회를 부를 때 인자로 넘기는 것이 아니라 조회
진입점을 만들 때 묶는다. 범위가 묶이지 않은 조회 진입점은 배포 단위 밖으로 노출되지 않는다.
사용자 범위가 없는 실행 요청은 접수되지 않으며, 기본 사용자로 떨어지는 경로를 두지 않는다.

## 근거

도구의 선언과 실행이 서로 다른 언어에 있으면 소유자가 없다. 한쪽은 인자를 검증하지만 그
인자로 무슨 일이 일어나는지 모르고, 다른 쪽은 실행하지만 모델에게 무엇을 약속했는지 모른다.
같은 검증 규칙이 두 벌 생기는 것은 그 결과지 원인이 아니다.

두 실행 백엔드를 나란히 두는 목적은 실행 방식의 차이를 관찰하는 것이다. 한쪽이 다른 쪽의
프로세스 메모리에 있는 함수를 빌려 쓰면 관찰 대상이 둘이 아니라 하나가 된다.

뷰를 계약으로 두는 이유는 결합 지점을 한 곳으로 모으기 위해서다. 파이썬이 기반 테이블을 읽으면
컬럼 이름 하나가 바뀔 때 그 사실이 조용히 건너간다. 뷰를 읽으면 같은 변경이 뷰 정의에서 먼저
막히고, 막힌 자리가 곧 고쳐야 할 자리다. 뷰는 파이썬이 무엇에 의존하는지를 한 파일에 적어 둔
목록이기도 하다.

사용자 범위를 생성 시점에 묶는 이유는 빠뜨릴 자리를 없애기 위해서다. 조회마다 조건을 하나 더
붙이는 규율은 언젠가 한 번 어긋나고, 그 한 번이 다른 사용자의 원장을 노출한다. 필수 인자는
빠뜨림은 막지만 잘못된 값을 넣는 것은 막지 못한다. 범위가 이미 묶인 진입점만 손에 쥐어 주면
다른 사용자를 고를 수단 자체가 없다.

모델이 사용자를 고르지 못하게 하는 것은 이와 별개의 층이다. 도구 인자 스키마에 사용자 범위를
두지 않고 스키마 밖 인자를 거부하면, 모델의 출력 공간에 그 값이 존재하지 않는다.

## 구현

### 뷰가 잠그는 것

뷰는 도구가 실제로 읽는 열만 노출한다. 열을 넓게 열어 두면 파이썬이 계약 밖의 열에 기대게
되고 계약이 도로 무의미해진다.

모든 뷰는 `userId`를 열로 갖는다. 사용자 범위가 조인을 거쳐야 하는 테이블은 뷰가 그 조인을
흡수해서, 파이썬 쪽 질의가 `userId` 하나로 범위를 좁힐 수 있게 한다.

뷰도 엔티티가 소유한다. 뷰 정의는 뷰 엔티티로 적고 마이그레이션은 거기서 생성한다. 손으로 쓴
정의문을 마이그레이션에 직접 넣지 않는다. 스키마의 단일 소유자가 엔티티라는 사실이 뷰에도
그대로 적용된다.

뷰는 살아 있는 행의 조건을 흡수한다. 삭제 표시와 검토 상태와 숨김 표시처럼 "이 행이 지금
보이는가"를 정하는 조건은 뷰 안에 있어야 한다. 그 조건을 파이썬이 다시 쓰면 두 곳이 어긋날 수
있고, 어긋나는 방향이 하필 지워진 것을 보여주는 쪽이다.

### 뷰가 잠그지 않는 것

집계와 요약은 뷰가 아니라 읽는 쪽이 만든다. 뷰는 행을 주고, 무엇을 세고 무엇을 상위 몇 개로
자를지는 도구가 정한다. 집계를 뷰에 넣으면 도구마다 다른 뷰가 필요해지고 뷰의 수가 도구의
수를 따라간다.

전문 검색은 뷰 밖이다. 색인이 원장과 다른 저장소이므로 계약도 다르다.

### 파이썬 쪽 형태

읽기는 사용자 범위가 묶인 조회 진입점이 소유한다. 진입점은 실행을 조립할 때 범위와 함께
만들어지고, 도구 핸들러는 그것만 받는다. 범위를 인자로 받는 공개 함수를 두지 않는다.
질의문을 만드는 코드는 그 진입점 안에 숨고 도구 핸들러는 질의문을 쓰지 않는다.

에이전트 슬라이스는 자기 도구의 실행을 자기 안에서 소유한다. 세 에이전트가 조회 코드를 공유하지
않는다.

### 사전 필터

모델을 부르기 전에 기계가 후보를 좁힐 수 있으면 좁힌 결과를 실행 요청 본문에 싣는다. 이것은
근거를 프롬프트에 미리 붙이는 것과 다르다. 후보가 비어 있으면 유료 호출을 아예 하지 않기 위한
것이고, 모델은 그 후보 안에서 무엇을 더 읽을지 여전히 스스로 고른다.

### 검색

전문 검색은 색인을 직접 질의한다. 색인은 문서 형태가 느슨하므로 뷰에 해당하는 계약을 색인
이름과 질의에 쓰는 필드 목록이 대신한다. 사용자 범위는 질의의 필터가 강제한다.

## 결과

- 도구를 하나 더하면 파이썬 안에서 끝난다. 다른 배포 단위를 고치지 않는다.
- 파이썬이 읽는 열이 늘어나면 뷰를 먼저 넓힌다. 뷰에 없는 열은 읽을 수 없다.
- 기반 테이블의 열 이름을 바꿔도 뷰가 흡수하면 파이썬은 모른다. 흡수할 수 없으면 뷰가 깨지고,
  두 배포 단위를 함께 고쳐야 한다는 사실이 그 자리에서 드러난다.
- 파이썬 배포 단위는 원장 데이터베이스와 검색 색인에 대한 자격 증명을 갖는다. 쓰기 권한은 갖지
  않는다.
- 사용자 범위가 비어 있는 실행 요청은 접수되지 않는다.
