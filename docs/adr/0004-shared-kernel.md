# ADR-0004 — 배포 단위는 서로를 모르고 공유 커널만 함께 쓴다

## 상태

채택됨

## 결정

서버, 웹, 에이전트 플러그인, 파이썬 에이전트는 서로를 직접 import하지 않는다. 예외를 두지 않는다.
넷 다 `packages/kernel` 하나만 공유하고, 커널은 어느 배포 단위도 알지 않는다.

커널은 데이터 전송 객체 라이브러리가 아니다. 경계를 넘는 값의 어휘와 스키마뿐 아니라 순수
도메인 규칙도 소유한다. 규칙 판정과 레시피 매칭이 그 예다.

## 근거

배포 단위가 서로를 import하기 시작하면 그 순간부터 따로 배포할 수 없다. 우산 경계에 예외를
하나 뚫으면 그 예외를 유지하기 위한 규칙이 여러 개 붙고, 규칙이 붙는다는 것은 구조가 그
예외를 못 견딘다는 뜻이다.

동시에, 순수 도메인 규칙까지 배포 단위마다 복사하면 판정이 갈라진다. 로컬 플러그인의 가드레일은
네트워크 없이도 서버와 같은 판정을 내야 한다. 서버가 규칙을 허용했는데 플러그인이 막으면
사용자는 시스템을 신뢰할 수 없다. 그러므로 판정 로직이 커널에 있는 것은 부채가 아니라 필연이다.

이름이 "계약"이면 데이터 전송 객체만 담아야 한다는 오해가 생기고, 그 오해를 단속하기 위한
규칙이 또 붙는다. 이름이 실체를 말하면 규칙이 줄어든다.

## 결과

- 커널은 프레임워크와 ORM에 의존하지 않는다.
- 커널의 스키마 검증기는 파일명이 `*.schema.ts`인 모듈에만 존재한다.
- 에이전트 플러그인과 웹은 커널의 스키마를 값으로 import하지 못한다. 타입으로만 쓴다.
  두 번들이 자립해야 하기 때문이다.
- 도메인 판단에 문자열 리터럴을 직접 비교하지 않는다. 어휘 상수를 커널에서 가져다 쓴다.
- 어휘의 복사본을 어디에도 두지 않는다. 파이썬 쪽도 커널의 값을 생성해 쓰거나 경계에서만
  문자열을 다룬다.
