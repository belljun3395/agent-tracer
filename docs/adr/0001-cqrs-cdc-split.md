# ADR-0001 — 쓰기와 읽기를 데이터베이스 수준에서 가른다

## 상태

채택됨

## 결정

쓰기 경로와 읽기 경로가 데이터베이스를 공유하지 않는다.

에이전트가 보내는 이벤트는 쓰기 전용 API가 받아 원장 데이터베이스에 추가만 한다.
변경 데이터 캡처가 이 원장을 스트림으로 흘리고, 투영기가 그 스트림을 소비해 읽기 데이터베이스의
읽기 모델을 갱신한다. 조회 API는 읽기 데이터베이스만 본다.

쓰기 측은 읽기 모델의 스키마를 알지 못한다. 코드 수준에서도 import하지 못한다.

## 근거

에이전트의 훅은 사용자의 작업 흐름 위에 얹혀 있다. 수집이 느려지면 사용자가 기다린다.
그래서 쓰기 경로는 추가만 하고 아무것도 계산하지 않아야 한다.

반대로 대시보드가 원하는 것은 계산된 형태다. 태스크의 현재 상태, 턴의 묶음, 규칙의 판정 결과는
전부 원시 이벤트에서 파생된다. 이 파생을 조회 시점에 하면 조회가 느리고, 쓰기 시점에 하면
수집이 느리다. 파생을 별도 프로세스로 밀어내면 양쪽이 각자의 속도를 갖는다.

원장이 진실이고 읽기 모델은 파생물이라는 관계가 명확하면 읽기 모델을 언제든 버리고 원장에서
다시 만들 수 있다. 이 성질이 구조 변경의 비용을 크게 낮춘다.

## 결과

- 이벤트는 추가만 되고 갱신되거나 삭제되지 않는다. 원장에는 서버가 매기는 단조 증가 시퀀스가 있다.
- 투영기는 마지막으로 적용한 시퀀스를 기록한다. 재시작하면 그 지점부터 이어서 소비한다.
- 읽기 모델은 원장에서 통째로 재생성할 수 있어야 한다. 재생성 명령이 있고, 재생성 결과가
  기존과 같은지 검증할 수 있어야 한다.
- 읽기 모델에 사용자가 직접 입력한 값(이름 변경, 숨김 처리 같은 것)을 섞지 않는다. 섞으면
  재생성이 그 값을 지운다. 사용자 입력은 원장에 이벤트로 들어가거나 별도의 테이블에 산다.
- 조회 API가 새 필드를 원하면 투영기가 그것을 만들어 두어야 한다. 조회 시점의 조인으로
  때우지 않는다.
