# ADR-0036 — 달러가 유일한 비용 통제 통화다

## 상태

채택됨

## 결정

AI 작업 실행의 종료를 가르는 통화는 달러 하나다. SDK 네이티브 비용 상한과 전역 모델 턴 상한이
실행을 막는 전부다. 도구 호출 개수는 어느 백엔드도 세지 않는다.

오케스트레이터가 워커나 후보에게 조사를 나눠 맡길 때 쓰는 값은 라운드가 아니라 weight다. weight는
서로 사이의 상대적 우선순위를 나타내는 정규화 비율일 뿐이며, 나눠 가질 고정 총량을 갖지 않는다.
런타임은 달러 상한에 `weight ÷ 전체 weight 합`을 곱해 각자의 비용 몫을 정한다. 비율 나눗셈이라
한 쪽에 남는 몫을 다른 쪽에 되돌려 주는 조정이 필요 없다.

전역 모델 턴 상한은 예산이 아니라 폭주만 끊는 느슨한 그물이다. 오케스트레이터와 워커 모두
같은 상한을 공유하며, 실제로 조사를 얼마나 깊이 파고들지는 각자에게 배분된 달러 몫이 정한다. 모델은 자기가
쓴 달러를 읽지 못하므로, 종료를 가르는 진짜 게이트와 별개로 지금까지 쓴 턴 수를 매 턴 알려줘
스스로 페이스를 조절하게 한다. 이 통지는 self-pacing을 위한 신호일 뿐 종료를 결정하지 않는다.

실행이 예산에 다다르면 하드 중단 대신 그때까지 모은 근거로 부분 결과를 낸다. 조사 도구를 거두고
결론 하나만 받는 마지막 호출로 넘어가는 방식이며, 두 실행 백엔드가 같은 결과를 낸다. 비용을 직접
세는 백엔드는 누적 지출에 가장 비쌌던 한 번의 호출 비용을 더해 다음 호출을 감당할 수 있는지로
이 전환점을 스스로 판정한다. 공급자 옵션으로 비용 상한을 거는 백엔드는 같은 판정을 자체 러닝
코스트 추정으로 근사하고, 공급자의 하드 상한은 그대로 최후의 안전장치로 둔다.

## 근거

도구 호출 개수는 비용 가드가 아니었다. 비용은 이미 달러가 막고, 하류 I/O 폭주는 도구 호출 수가
아니라 그 폭주가 실제로 일으키는 문제(연결 소진, 쓰기 폭주) 지점에서 막는 편이 정확하다. 두
백엔드가 도구 호출 개수를 각자 다른 스코프(조사자 인스턴스마다 대 실행 전체마다)로 세면 그
차이 자체가 비교 대상을 실행에서 집행 규칙으로 옮겨, 두 백엔드를 나란히 두고 실행 방식만 비교
하려는 목적을 해친다.

라운드는 두 얼굴을 가졌다. 배분 통화로서의 라운드와 폭주 방지로서의 라운드다. 배분은 weight로
충분하다. 오케스트레이터가 상대적으로 얼마나 공을 들일지만 정하면 되고, 그 비율을 달러에
곱하면 배분이 끝난다. 고정된 라운드 풀을 먼저 나누고 남는 만큼 최소 몫을 보장하다가 모자라면 비례로 깎는 계산은
이 비율 나눗셈 앞에서 군더더기였고, 그 군더더기가 배분 규칙을 실행 규칙보다 복잡하게 만들었다.

하드 중단은 그 시점까지 모은 근거를 통째로 버린다. 예산이 빠듯한 실행일수록 근거는 이미 많이
쌓여 있고 버릴 이유가 없다. 도구만 거두고 결론 턴 하나를 더 받으면 그 근거가 그대로 산출물이
된다. 비용을 공급자 옵션으로 위임한 백엔드도 이 이득을 놓칠 이유가 없어, 자체 추정으로 같은
전환점을 근사한다. 추정이 어긋나도 공급자의 하드 상한이 마지막 안전장치로 남는다.

## 결과

- 도구 호출 개수를 세는 상한을 다시 들이지 않는다. 하류 I/O 폭주가 실제 문제라면 그 지점에서
  별도로 막는다.
- 오케스트레이터의 배분 스키마는 weight를 받는다. weight의 상한은 배분 비율의 상한일 뿐이며, 워커
  인스턴스의 턴 백스톱을 정하는 상수와는 별개다.
- 구독 인증처럼 토큰 단가를 근사로만 아는 실행 경로에서도 달러는 여전히 유효한 예산 신호다.
  근사가 정밀하지 않다는 사실을 받아들인다.
- 누적 비용으로 하는 착지 예측은 근사이므로 공급자 네이티브 비용 상한을 대체하지 않고 그 앞에서
  선제적으로만 작동한다.
