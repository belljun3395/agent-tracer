# ADR-0026 — 에이전트마다 백엔드가 둘이고, 남는 차이는 실행 기계의 차이뿐이다

## 상태

채택됨

## 결정

AI 잡 에이전트는 셋이다. recipe-scan, title-suggestion, task-cleanup이다.
셋 다 백엔드가 둘이다. 워커가 Claude Agent SDK로 직접 실행하는 것과 파이썬 그래프에 위임하는
것이다. 어느 쪽을 쓸지는 잡 입력이 정하고, 없으면 앱 설정의 기본값을 쓴다.

두 백엔드가 각자 자기 프롬프트와 도구와 출력 모델과 검증 정책을 갖는다. 같아야 하는 사실은
커널의 계약 픽스처가 소유하고 두 언어의 테스트가 그 파일을 읽는다.

세 에이전트가 공통으로 따르는 실행 형태는 하나다.

**조사 → 검증 → (한 번) 수리 → 확정 또는 빈 결과.**

모델이 도구를 골라 근거를 모으고, 결정적 검증기가 그 출력이 이 실행에서 관측한 근거만
인용하는지 검사하고, 어긋나면 오류를 모델에게 돌려주고 한 번만 다시 받는다. 그래도 근거가
서지 않으면 버린다. 빈 출력은 옳은 답이므로 오류가 아니다.

## 근거

두 백엔드를 나란히 두는 목적은 실행 방식의 차이를 관찰하는 것이다. 지식이 갈라지면 관찰의
대상이 실행이 아니라 지식이 되어 실험이 무의미해진다. 그래서 지식은 계약이 잠그고, 실행
기계는 자유롭게 둔다.

검증과 수리를 두 백엔드가 공유해야 하는 이유는 따로 있다. 모델이 지어낸 식별자를 그대로
저장하면 사용자가 근거를 눌렀을 때 아무것도 없다. 한쪽 백엔드만 그것을 걸러내면 같은 제품이
백엔드 운에 따라 다른 품질을 낸다.

## 구현

### 계약이 잠그는 것

`packages/kernel/src/agent/__fixtures__/*.tool.contract.json`이 에이전트마다 하나씩 있다.
두 언어의 계약 테스트가 그 파일을 읽는다.

- 도구 이름과 인자의 필수 여부와 기본값과 상하한과 열거값.
- 모델이 읽는 도구 설명 문장.
- 도구 응답 본문의 필드.
- 출력 스키마의 필드와 상한.
- 예산. 턴 한도, 토큰 한도, 비용 한도, 후보 개수 한도.

한쪽 구현만 고치면 그쪽 테스트가 깨지고, 계약만 고치면 반대쪽이 깨진다.

### 에이전트별 형태

**recipe-scan.** 도구 여섯(태스크 요약, 이벤트 페이지, 규칙 목록, 이벤트 검색, 유사 태스크,
레시피 검색). 근거 장부가 도구가 돌려준 taskId·turnId·eventId·ruleId·레시피 개정 번호를 모은다.
검증기는 앵커 태스크 인용, 관측한 식별자만 인용, 같은 턴을 두 후보가 주장하지 않을 것,
스텝 번호의 연속성을 본다. 검증에 걸리면 후보 목록 전체를 다시 받는다.

**title-suggestion.** 도구 하나(이벤트 페이지). 검증기는 개수와 현재 제목 반복과 후보 중복과
자리표시자 제목(untitled, test, task 3 따위)을 본다. 근거 인용이 없는 에이전트라 장부가 없다.

**task-cleanup.** 도구 둘(정리 후보 목록, 이벤트 페이지). 근거 장부가 목록에 오른 후보와
실제로 열어본 태스크의 이벤트를 모은다. 검증기의 핵심은 하나다. **이벤트가 있는 태스크는
열어보고 그 이벤트를 인용해야 보관을 제안할 수 있다.** 검증에 걸린 제안 하나만 버리고 나머지는
살린다. 인용한 이벤트 식별자는 합격 기준을 위한 값이므로 저장하지 않는다.

### 남는 차이 — 실행 기계의 차이

계약이 잠그지 않고, 잠글 이유도 없는 차이다.

- **수리를 다시 묻는 방법.** SDK는 대화를 잇지 않으므로 직전 출력을 프롬프트에 다시 싣는다.
  파이썬은 살아 있는 메시지 목록에 지시문을 덧붙인다.
- **프롬프트의 문장.** 말해야 하는 사실은 같지만 문장은 각자의 것이다. 도구 설명은 여기서 빠진다.
  모델이 무엇을 할 수 있는지 알려 주는 문장이라 갈라지면 비교 대상이 실행이 아니라 정보가 된다. 파이썬은 프롬프트
  버전 문자열을 프롬프트에 실어 궤적에서 의미 변화의 경계를 식별한다.
- **예산 집행 지점.** SDK는 공급자 옵션으로 비용 한도를 건다. 파이썬은 단가 근사표로 직접
  세어 넘으면 실행을 끊는다. 그래서 파이썬은 단가표에 없는 모델 이름을 만나면 실행을 거부한다.
- **폴백 모델과 추론 강도.** SDK 경로에만 있다.
- **도구 루프의 여유.** 파이썬은 스키마에 맞지 않는 출력을 한 번 더 요청하는 재시도를 도구
  루프 안에 갖는다. 라운드 예산을 쓰고 나서 마지막으로 한 번 더 부른다.
- **관측.** SDK는 워커의 텔레메트리 래퍼를 통과하고, 파이썬은 OTel 스팬과 실행 궤적을 낸다.
  둘 다 스텝 단위 궤적을 남기며 형태가 같아야 비교가 된다.
- **조사의 분해.** 파이썬은 조건부 엣지의 `Send`로 전문가를 명시적으로 띄우고, SDK는 부모 모델이
  네이티브 `Agent` 도구로 역할별 서브에이전트를 호출한다. 파이썬은 fan-out과 예산 몫을 코드가
  결정하지만 SDK는 어느 전문가를 언제 부를지 부모 모델이 결정한다. title-suggestion처럼 분해의
  이득이 없는 짧은 실행은 양쪽 모두 단일 조사 루프를 유지한다.
- **그래프 위상.** 파이썬은 LangGraph 노드와 조건부 엣지로 위 형태를 그리고, SDK는 어댑터의
  코드로 그린다.

이 차이들은 최종 출력과 성공률과 근거 정확도와 비용과 지연에 드러난다. 비교의 대상이 바로
그것이다.

## 결과

- 새 도구나 새 출력 필드나 새 예산을 더하려면 계약 픽스처를 먼저 고친다. 두 언어의 테스트가
  동시에 빨개지고 둘 다 고쳐야 초록이 된다.
- 프롬프트를 고칠 때는 그 사실이 계약에 있는지 본다. 계약에 있으면 양쪽 프롬프트가 함께 말한다.
- 한 백엔드에만 있는 검증 규칙을 만들지 않는다. 그것이 곧 갈라짐이다.
- 도구 응답에 필드를 더해도 백엔드가 죽지 않는다. 모르는 필드는 무시하고 계약이 형태를 지킨다.
