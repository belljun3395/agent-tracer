  • DB 테이블 event_files: 파일 경로 ↔ 이벤트 매핑 + write_count — packages/server/src/main/migrations/1700000003000 L36–45 (idx_event_files_path 인덱스 존재)
  • DB 테이블 event_relations: 이벤트 간 엣지 — edge_kind(parent/source/related), relation_type(implements / revises / verifies / answers …) 컬럼 이미 존재 (L47–61)
  • 도메인 엔티티: EventFileEntity, EventRelationEntity — packages/server/src/activity/event/domain/

부족한 것

  • Read → Edit → Write 의존 체인을 자동으로 추론하는 로직 (현재 relation_type 컬럼은 비어 있음)
  • 파일 기준 역방향 추적용 그래프 쿼리 엔드포인트 (현재 timeline 기준만)

재사용 인프라

  • event_relations 스키마 그대로 + relation_type 채우는 백필 job 하나면 됨
  • idx_event_files_path 활용 → 파일 한 개로 모든 접근 이벤트 즉시 조회
Case Study2026년 현재 'AI 코딩용 KG'는 가장 뜨거운 영역 중 하나
  • code-review-graph — Claude Code 전용 로컬 KG. 데몬이 파일 변경 감시 → 점진 재구축. 리뷰 토큰 6.8× / 일상 작업 49× 절감 주장.
  • GitNexus (MarkTechPost 2026-04-24) — MCP 네이티브 KG 엔진. 진입점에서 콜체인을 추적해 "processes" 노드 생성. Claude Code / Cursor 컨텍스트 주입용.
  • Graphify / CodeGraphContext — Tree-sitter AST 기반. 코드 + DB 스키마 + 인프라 한 그래프.

공통 패턴

  • Tree-sitter 로 정적 AST 그래프 → MCP resource 로 agent 에 노출 → PostToolUse 로 재인덱싱 트리거.
  • Agent Tracer 만의 차별점: 위 도구들은 모두 정적 코드 그래프. "실제 agent 의 read/edit 시퀀스 자체"를 엣지로 쓰는 사례는 GitNexus 외엔 거의 없음 → 빈 영역.
함정 — "Stale graph"
Karpathy 의 llm-wiki 메모: "업데이트가 미해결 과제. 소스 하나 추가가 entity·timeline·index 에 전파되며 충돌 해소·중복·provenance 가 깨진다. lossy compression 으로 caveat·날짜·edge case 누락." → 그래프는 만드는 것보다 최신 상태로 유지가 진짜 비용.
Official DocsPostToolUse hook 만으로 공식 채널 완결
  • Hook 이벤트: PostToolUse 가 정공법. tool_input 에 파일 경로, tool_response 에 성공 여부 포함. 대량 일괄 수집은 SessionEnd.
  • 회색 지대: tool_response 에는 실제 diff 가 안 옴. 보강안 → PostToolUse 훅에서 직접 git diff 를 부르거나 사전 스냅샷 비교.
"PostToolUse": [
  { "matcher": "Edit|Write|Read",
    "hooks": [{ "type": "http", "url": "localhost:3847/file-hook" }] }
]

이미 Agent Tracer 가 PostToolUse 를 쓰고 있어 hook payload 만 그래프 빌더로 fan-out 하면 됨.

Viability"그래프"라는 형태가 진짜 가치인가에 의문

Use case

  • 세션 1개 안의 파일 리스트는 이미 task-list / inspector 로 충분 → 같은 파일을 반복적으로 만지는 hotspot 발견 같은 cross-session 패턴에서만 의미 있음.

정체성 정합

절반만 맞음. observer 본업의 자연스러운 확장이지만, force-directed 그래프 같은 화려한 시각화에 시간이 빨려나가는 신호가 됨.

ROI / 함정

  • 데이터 추출 S, 의미 있는 시각화/쿼리 L. 가치 중간.
  • 그래프는 노드 100개 넘으면 사람이 못 읽음 → 결국 "표 + 필터"로 회귀.
권고 — 그래프 대신 hotspot 테이블로 시작
"파일 hotspot 테이블 + 관련 세션 링크" 라는 작은 형태로 출발하고, 가치 검증된 후에 그래프 시각화로 확장.

실무 함정

Viability가장 비싸고, 가장 안 쓰이게 될 가능성이 큰 항목

Use case

  • "외출 중 휴대폰에서 진행 중인 agent 에 추가 지시" — 1인 개발자 + 로컬 도구 컨텍스트에서 use case 가 약함.
  • 책상 앞에 있을 때 99% 작업하므로 "있긴 한데 안 씀"의 교과서가 될 위험.

정체성 정합

어긋남. observer → controller 로 정체성이 바뀜. 인증·터널링·세션 충돌·CORS·보안 모델까지 새로 설계해야 함.

ROI

난이도 XL, 가치 낮음~중간. 1인 프로젝트가 감당하기엔 surface area 가 과대.

결론 — 만들지 않는 쪽 권고
정 원하면 "읽기 전용 원격 모니터링"(현 대시보드를 ngrok 으로 노출)까지만. 쓰기/제어는 Anthropic 공식 Remote Control 에 위임.
