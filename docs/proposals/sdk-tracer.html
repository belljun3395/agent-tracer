Agent Tracer · Tier 5 detailed design

@monitor/sdk-tracer — 다른 SDK 사용자를 위한 계측 라이브러리

Agent Tracer는 현재 Claude Code 플러그인과 Codex hook 어댑터만 자동 계측한다. Anthropic @anthropic-ai/claude-agent-sdk 로 직접 만든 사내 봇·평가 하니스·고객 대상 AI 기능은 수동 HTTP 호출로만 합류할 수 있다. @monitor/sdk-tracer는 SDK의 query()를 얇게 감싸서 같은 privacy contract · 같은 ingest 페이로드 · 같은 대시보드 surface를 한 줄 교체로 가져가게 한다.

1 풀고 싶은 문제

Claude Code의 훅은 “사용자가 IDE에서 직접 쓰는 인터랙티브 세션”에만 붙는다. 그런데 실제 조직에서는 SDK를 통해 훅이 닿지 않는 환경에서도 Claude가 돌아간다 — 슬랙 봇, 야간 batch, CI 안의 코드 리뷰어, 고객 채팅에 임베드된 에이전트. 이들에 대해서도 “정확히 어떤 동작을 했나” “룰을 지켰나”를 같은 도구로 보고 싶다.

이 격차는 Agent Tracer의 가장 큰 채택 장벽이기도 하다. Claude Code 한정 도구로 보이면 SDK 기반 프로덕션 워크로드를 운영하는 팀은 다른 솔루션을 찾는다.

2 유즈케이스

각각의 케이스는 모두 “SDK 환경에서 Agent Tracer를 그대로 쓰고 싶다”를 다른 옷으로 입은 것.

UC-A

Slack 봇 — 사내 PM이 “요청 사항을 PR로 만들어줘”

Long-lived process

Express + Slack Bolt 위에 query()로 짠 봇. 동일 channel-thread를 같은 session으로 묶어 멀티-턴 대화 유지. 운영 중 “왜 봇이 user repo의 잘못된 파일을 만졌지?”가 자주 발생.

현재
로그를 grep으로 뒤짐. tool_use가 정형화돼 있지 않아 어떤 파일을 만졌는지 한눈에 모름.
tracedQuery 적용 후
슬랙 thread = task. 대시보드에 자동 등록. 룰 “.env 수정 금지”가 Claude Code 세션과 동일하게 작동.
UC-B

평가(eval) 하니스 — 한 프롬프트를 1,000번 돌려 모델 비교

Batch

ML 팀이 nightly로 “refactor 100개 후보 prompt”를 claude-haiku-4-5와 claude-sonnet-4-6로 각각 돌린다. 한 모델이 다른 모델보다 “덜 파일을 만진” 케이스를 찾고 싶음.

현재
결과 JSON을 별도 노트북에서 집계. 룰 위반/통과는 수동 분석.
tracedQuery 적용 후
매 평가 = 1 task. 같은 룰 세트 + Judge Loop가 자동 평가 → 모델 간 verdict 분포 비교가 대시보드 차트.
UC-C

CI 안의 PR 리뷰어 — GitHub Action 컨테이너

Ephemeral

PR 열릴 때 컨테이너 안에서 query()로 코드 리뷰 코멘트를 생성. 컨테이너는 분 단위로 사라지고, 로그는 휘발됨.

현재
PR 코멘트만 남고 “리뷰어가 어떤 파일을 살펴봤는지”는 사라진다. 룰 적용 불가.
tracedQuery 적용 후
컨테이너에서 MONITOR_BASE_URL로 사내 모니터에 push. PR ID = task ID. 사후 “이 리뷰어가 PR 외 파일을 봤나?” 룰 검증.
UC-D

고객 대상 인앱 에이전트 — Production observability

Production

사용자 “내 데이터 분석 좀”에 응답하는 인앱 어시스턴트. 다수의 동시 세션, 각 세션은 분~시간 단위.

현재
에러율과 latency만 APM에 있음. “어떤 query를 어떤 순서로 실행했는지”는 부분 노출.
tracedQuery 적용 후
user_id → task tag로 매핑. 룰: “고객 PII 컬럼을 평문 출력 금지”. 위반 시 PreToolUse 차단 로직과 동일한 의미의 사후 알람.

3 아키텍처

핵심 아이디어: 사용자는 평소처럼 SDK 메시지 스트림을 받는다. tracer는 같은 스트림을 옆에서 한 번 더 듣고 이벤트 변환기로 ingest payload를 만들어 모니터에 비동기 POST. 사용자 코드 입장에선 SDK가 똑같이 동작하고, 네트워크 실패 시에도 메인 에이전트는 절대 막히지 않는다.

그림 1. 사용자 코드는 SDK 메시지 스트림을 받는다(투명 전달). tracer는 옆에서 같은 스트림을 듣고 변환·전송한다.

4 이벤트 변환 표 — SDK 메시지 → ingest payload

이 표가 핵심 산출물. SDK가 내는 모든 메시지 종류에 대해 “무엇을 보내고” “무엇을 버리는지”를 정의한다. Codex 어댑터의 action-only 정책과 동일.

SDK 메시지 매핑되는 ingest 전송하는 필드 (kept) 버리는 필드 (dropped)
system.init POST /ingest/v1/sessions/ensure + tasks/start session_id, model, cwd, tools_available[] —
user (text) POST /ingest/v1/conversation {kind:"user.message"} prompt 텍스트 (raw 캡처 ok) —
user (tool_result) — (서버 저장 안 함) tool_use_id (상관관계용) result.content 전체
assistant (text) POST /ingest/v1/conversation {kind:"assistant.response"} message text —
assistant (thinking) events {kind:"context.saved", lane:"planning"} thinking 요약 (옵션) signature 등 내부 메타
tool_use (Bash) tool-activity {kind:"terminal.command"} command, description, timeout env, stdin
tool_use (Read/Edit/Write) tool-activity {kind:"tool.used"} file_path, offset/limit, edit count old_string, new_string 본문
tool_use (Grep/Glob) tool-activity (exploration lane) pattern, path, glob, output_mode matched line content
tool_use (WebFetch/WebSearch) tool-activity (exploration lane) url(s), query, allowed_domains response body 전체
tool_use (mcp__*) coordination tool name, input keys 민감 인자 (자동 마스킹)
tool_use (Agent) coordination + tasks/link subagent_type, description subagent prompt 본문
result sessions/end + telemetry stop_reason, usage(token), latency, cost —

ⓘ “kept”는 기존 Claude Code 훅이 캡처하는 것과 같은 집합. 즉 SDK 경로에서도 같은 데이터 모양이 나오므로 Inspector / Rules / Judge 모두 별도 분기 없이 동작.

5 Public API 디자인

두 가지 진입점. tracedQuery는 1회용·짧은 호출에, tracedSession은 대화형·장기 세션에.

// packages/sdk-tracer/src/index.ts
export interface TracedQueryOptions extends SDKQueryOptions {
  task?: {
    id?: string;
    title?: string;          // 기본: prompt 첫 80자
    source?: string;         // 기본: "sdk-app"
    tags?: Record<string, string>;
    parentTaskId?: string;
  };
  monitor?: {
    baseUrl?: string;        // 기본: env MONITOR_BASE_URL → http://127.0.0.1:3847
    apiKey?: string;
    failureMode?: "silent" | "throw";   // 기본 silent
    flushOnExit?: boolean;   // 기본 true
  };
  ruleSync?: {
    inject?: boolean;        // 활성 룰을 system prompt로 주입
    pollIntervalMs?: number;
  };
}

export async function* tracedQuery(
  opts: TracedQueryOptions
): AsyncGenerator<SDKMessage> { /* yield 사용자에게, side-channel 전송 */ }

export class TracedSession {
  send(prompt: string): AsyncGenerator<SDKMessage>;
  checkpoint(label: string): Promise<void>;
  end(): Promise<void>;
  readonly taskId: string;
  readonly sessionId: string;
}

export function tracedSession(opts: TracedQueryOptions): TracedSession;

7 장애 격리 — 모니터가 죽어도 에이전트는 산다

Production 채택의 마지노선은 “계측이 본체를 죽이지 않을 것”. fail-open + circuit breaker + bounded queue 3겹.

그림 3. 4-상태 머신. 모든 상태에서 “사용자 SDK 호출의 latency 추가 = 0” 보장.

11 사용 예제

UC-A · Slack 봇

import { App } from "@slack/bolt";
import { tracedSession } from "@monitor/sdk-tracer";

const bolt = new App({ token: env.SLACK_TOKEN, signingSecret: env.SLACK_SECRET });
const sessions = new Map<string, ReturnType<typeof tracedSession>>();

bolt.message(async ({ message, say }) => {
  const key = message.thread_ts ?? message.ts;
  let s = sessions.get(key);
  if (!s) {
    s = tracedSession({
      task: { title: message.text.slice(0, 80),
              source: "slack-bot",
              tags: { channel: message.channel, user: message.user } },
      monitor: { failureMode: "silent" }
    });
    sessions.set(key, s);
  }
  for await (const ev of s.send(message.text)) {
    if (ev.type === "assistant" && ev.text) await say({ text: ev.text, thread_ts: key });
  }
});

12 리스크와 완화

SDK 메시지 스키마 변화

Anthropic이 SDK 메시지 모양을 바꾸면 translator가 깨짐. 완화: SDK 버전 매트릭스 + contract 테스트(실제 fixture 캡처). semver-major 표시.

privacy 누락 회귀

tool_use 새 종류 추가될 때 화이트리스트 미반영. 완화: snapshot 테스트로 “새 필드 등장 시 자동 빨강”. 기본 동작은 “모르면 drop”.

SDK 호환성

SDK 메이저 업그레이드 시 사용자가 둘을 동시에 올려야 함. 완화: peerDep으로 명시, 범위 좁힘, examples를 CI에서 실제 SDK 버전으로 빌드.

대량 trace로 모니터 과부하

UC-B처럼 1000건 동시 발생. 완화: 서버 측 ingest는 이미 batch + idle backoff(perf/queue 커밋). tracer는 50ms micro-batch로 1차 압축.

인증 / 멀티 테넌시

모니터는 현재 로컬 신뢰 모델. 사내/공용 배포 시 API key 필요. 완화: monitor.apiKey 옵션을 동시에 도입하고 서버 측 미들웨어 추가(별도 작업이지만 인터페이스는 미리).

OpenInference / OTEL 이중 export

OTEL bridge 활성 시 span 중복. 완화: tracer의 OTEL bridge는 opt-in이고, 서버 측 OpenInference export와 동일 trace_id 규약 사용해서 중복 검출 가능.
