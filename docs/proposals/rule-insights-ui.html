Agent Tracer · Design Proposal

Rule Insights UI — verdict & evidence 표면화 설계안

Rule 시스템과 verdict 인프라는 이미 데이터를 다 만들고 있는데, UI 가 그걸 graph/feed 의 lane 색 으로만 노출하고 있어서 사용자가 "에이전트가 제대로 일했나?" 라는 질문에 답하기 어려운 상태. 이 문서는 그 질문에 직접 답하는 6가지 UI 개선을 정리하고, 어느 순서로 만들지 결정합니다.

대상: packages/web · packages/server (얇은 endpoint 추가 정도) 작성일: 2026-05-12 전제: rule-enhancement.html (생성 인프라) 구현 완료 후

A문제 정의 — 지금 무엇이 비어 있는가

현재 상태:

  • Rule 매칭 데이터는 rule_enforcements (eventId, ruleId, matchKind) 에 다 적힘
  • Verdict 도메인(governance/verification) 도 trigger/expect 매칭을 turn 단위로 집계
  • 그런데 UI 는 그래프와 피드에서 lane 색만 바꿔서 표시

그 결과:

  • "agent 가 lint 돌렸나?" — timeline 을 직접 스크롤해서 사용자가 머릿속으로 채점
  • "rule X 가 어떤 파일/명령을 근거로 봤나?" — graph 의 lane 변경 안에서 다른 이벤트들과 섞임
  • "내가 mention 한 파일 진짜 봤나?" — 비교 도구 없음
  • "rule 이 의도대로 작동하나?" — match 됐는지 안 됐는지 디버깅 단서 없음

핵심 진단: rule 인프라가 부족한 게 아니라, 이미 만들어진 신호의 노출 이 부족하다.

1Per-rule Evidence — Rules 탭 row 펼치기 ★ 사용자 선택 · 최우선

Lead 한 rule 이 "본" 모든 파일과 행위를 row 안에서 펼침 S+B (≈ 3일)

Inspector → Rules 탭의 각 rule row 를 클릭하면 펼침. 그 rule 의 증거 이벤트 가 Files / Actions / Missed 세 묶음으로 노출.

스케치

[v] Lint should be run                          fired 3x  v
    ─────────────────────────────────────────────────────
    Files (0)
      (no file references)

    Actions (3)
      ▶ Bash · npm run lint              14:23  trigger
      ▶ Bash · npm run lint -- --fix     14:24  expect
      ▶ Bash · pnpm lint                 14:31  expect

[v] Read auth modules before editing             fired 5x  v
    ─────────────────────────────────────────────────────
    Files (3)
      📖 src/auth/login.ts          read 14:20  expect
      📖 src/auth/utils.ts          read 14:21  expect
      ✏️ src/auth/middleware.ts     edit 14:22  expect

    Actions (5)
      Read · 3x        Edit · 2x

[v] Mention should be read                       fired 2x  ⚠ partial
    ─────────────────────────────────────────────────────
    Files (mentioned 4 · matched 2)
      v src/auth/login.ts         read 14:20
      v src/auth/utils.ts         read 14:21
      x docs/auth-spec.md         not seen
      x tests/auth.test.ts        not seen

인터랙션

  • 각 행 클릭 → timeline 의 해당 event 로 점프 + 잠깐 하이라이트
  • 매 row 우측에 trigger/expect 표시 → 어느 쪽 조건으로 매치됐는지 즉시 보임
  • "Missed" 묶음 (mention 등) 은 같은 패널 안에 ✗ 로 함께 노출 — 따로 화면 안 옮겨도 됨

왜 이게 lead 인가

  • 사용자가 직접 "이게 가장 좋다"고 합의함
  • 새 페이지/탭 없음 — 이미 보고 있는 Rules 탭에서 그대로 펼침
  • "rule 의 근거 + 누락" 이 한 자리에 모임 → 디버깅과 검증을 동시에

데이터 / API

  • 새 endpoint: GET /api/v1/tasks/:taskId/rules/:ruleId/evidence
  • 또는 기존 useTaskRulesQuery 의 응답에 evidence 를 함께 join — task 당 rule 수가 적으니 폭증 안 함
  • 응답 shape: { triggers: [...], expects: [...] } — 각 항목에 eventId, kind, title, command?, filePath?, matchedBy

2Task Header Rule Scorecard

Complement Task 화면 상단에 한 줄로 verdict 칩 S (≈ 1-2일)

한 task 의 전체 verdict 를 머리에 띠로 노출. 펼침 없이 바로 "이번 작업 잘 됐나" 가 답이 됨.

┌──────────────────────────────────────────────────────────┐
│ Rules: [v Lint]  [v Test]  [x Typecheck]  [⚠ DB Migrate] │
│        2 verified · 1 contradicted · 1 unverifiable     │
└──────────────────────────────────────────────────────────┘
  • 각 chip 클릭 → 해당 rule 의 evidence panel 펼치는 위치로 스크롤 (= ①과 자연스럽게 연결)
  • 색상: 초록 ✓ / 빨강 ✗ / 노랑 ⚠
  • 데이터는 verification 도메인의 verdict 가 이미 계산하고 있음 — endpoint 만 노출

①과 함께 만들면 시너지: scorecard (요약) → 칩 클릭 → evidence panel (드릴다운).

3"Missed" 표시 + matchedBy 라벨

Complement ①의 자연스러운 확장 — rule 디버깅 정보 XS+M (≈ 2일)

matchedBy 라벨 (XS)

각 매치된 이벤트에 어떤 조건으로 매치됐는지 라벨 표시:

▶ Bash · npm run lint  14:23  trigger        [via commandMatch]
▶ Bash · pnpm lint     14:31  expect         [via commandMatch]
📖 src/auth/login.ts   14:20  expect         [via action=file-read]
📖 docs/notes.md       14:22  expect         [via pattern]
  • rule 작성자가 "내 regex 오타 났나" 같은 디버깅 가능
  • 구현: matching engine 이 이미 어떤 조건이 매치됐는지 알고 있음 — 그 정보만 응답에 추가

Missed 표시 (M)

"trigger 는 매치됐는데 expect 가 같은 turn 안에서 매치 안 된 경우" 를 ✗ 로 같은 패널에 표시 (위 ① 예시의 "Mention should be read" 처럼). verdict 도메인의 contradicted 결과를 그대로 가져와 evidence 패널과 묶음.

Later Rule 무관, 그냥 task 활동량 요약 M (≈ 2-3일)

│ 📖 Read     ████████████ 12       │
│ ✏️ Write    ████ 4                │
│ ▶ Bash     ██████ 6               │
│ 🌐 Web     ▏0                     │
│                                   │
│ Top files (by touches)            │
│  src/auth/middleware.ts    ●●●    │
│  package.json              ●●     │
│  tests/auth.test.ts        ●      │

  • 데이터는 우리가 만든 monitor_get_task_summary 가 이미 계산 (toolCounts, topFiles, topCommands)
  • Overview 탭에 위젯 추가만 하면 됨
  • "이번 작업이 read-heavy 였나 write-heavy 였나" PR 회고용

6Timeline verdict overlay

Future Graph/Feed 노드에 ✓/✗ 배지 M+ (디자인 비용 큼) 
○ Read package.json           ← 일반
●v Bash: npm run lint         ← rule "Lint" fulfilled
●x (이 자리에 expected Bash 가 없었음)  ← contradicted 표시

지금 lane 색만 바꾸는 것보다 강한 신호지만, ①~③ 로 verdict 가 다른 곳에서 충분히 노출되면 후순위. 시각 디자인 복잡도가 큼.

D열린 질문

  1. evidence 응답 크기 — task 당 rule 수 × match 수가 폭증할 가능성. row 별 lazy fetch (펼침 시점에만) vs 처음부터 함께 — UX 와 비용의 트레이드오프. 일단 lazy 가 안전.
  2. matchedBy 라벨의 정확도 — matching engine 이 "어느 조건으로 매치됐는가" 를 정확히 한 가지로 답할 수 있나, 아니면 여러 조건 동시 매치되는 경우가 있나. 후자라면 멀티 라벨 필요.
  3. "Mention should be read" 같은 multi-path rule — 현재 rule 모델은 한 pattern 만 들고 있어서 4개 mention 검증을 한 rule 로 표현 어려움. ①의 "Files (mentioned 4 · matched 2)" 스케치를 진짜로 하려면 verdict 가 turn 단위로 다중 path 를 추적해야. 별도 설계 필요할 수 있음 — 일단 ①은 단일 path rule 만 지원하고 시작.
  4. verdict 의 "unverifiable" — 현재 verdict 모델에 "trigger 가 안 일어났다" vs "expect 못 채웠다" 가 구분되는지. Scorecard 의 ⚠ 가 어떤 케이스를 의미할지 명확히.
