Agent Tracer · Design Proposal

Rule 시스템 고도화 설계안 v2 · 슬래시 명령 우선

v1 의 핵심 두 가지를 뒤집습니다. ① setup-time CLI 프롬프트는 폐기 — 대신 Claude Code 슬래시 명령 /setup-rules 가 그 역할을 합니다 (파일 탐지 + 짧은 설문 결합). ② Phase B 의 LLM 호출은 단순 API 가 아니라 Claude Agent SDK (도구 포함) 입니다 — 서버가 spawn 한 에이전트가 task.workspacePath 를 cwd 로 두고 사용자 로컬 파일을 Read/Glob/Grep 으로 직접 탐색합니다.

대상: packages/runtime, packages/server, packages/web 작성일: 2026-05-12 (v2) 이전: v1 폐기 — CLI prompt + readline + seed file 경로 모두 제거

A멘탈 모델: SDK Agent 가 무엇이며 어디서 도는가

v1 에서 "서버가 Anthropic API 를 부른다" 고 적어서 마치 stateless 호출처럼 읽혔던 부분을 명확히 합니다.

호출 방식도구 (Read/Glob)인증어디서 도는가
Anthropic API 직호출 (v1 오해 소지) x 없음 — 텍스트 in/out API key 필요 호출자 프로세스 (stateless)
Claude Agent SDK (v2 선택) v Read/Glob/Grep/Bash 내장 API key 필요 호출자 프로세스 내부에 에이전트 spawn — cwd 옵션으로 파일 접근 범위 지정
Claude Code 슬래시 명령 v Claude Code 본체 도구 그대로 Claude Code 가 이미 인증 사용자가 띄운 Claude Code 세션 안 (= 사용자 cwd)
핵심 통찰
Agent Tracer 의 모니터 서버는 사용자 본인 머신 에서 돕니다. 서버 프로세스가 SDK Agent 를 spawn 하면, 그 에이전트는 같은 머신의 파일 에 접근 가능. task.workspacePath 를 cwd 로 넣으면 해당 프로젝트 디렉터리로 자연스럽게 스코프됨. "원격 서버" 같은 게 아니라 그냥 로컬 monitor 데몬이 띄운 서브-에이전트.

B재사용 자산 (스키마 무변경 영역)

Rule 도메인 자체는 v1 과 동일하게 손대지 않습니다.

  • Rule 스키마: source=agent, severity=info, scope=global, signature dedup 그대로
  • Rule 매칭 엔진: 실시간 enforcement + backfill 그대로 — 슬래시 명령으로 만든 rule 도 자동 적용
  • REST API: POST /api/v1/rules, re-evaluate, promote 그대로
  • MCP 인프라: packages/server/src/adapters/mcp/tools/ 에 이미 task/event/conversation 툴이 있으므로 같은 패턴으로 rule 툴만 추가
  • Plugin 구조: packages/runtime/.claude-plugin/plugin.json 이 이미 존재 — commands/ 디렉터리만 신설하면 됨

A-2. setup-rules.md 프롬프트 본문 (요지)

파일 탐지는 "현재 설정된 것" 만 알려주고, 사용자가 "중요하게 여기는 것" 은 모름. polyglot 프로젝트(예: package.json + Cargo.toml 공존), framework 별 관습, "언제" 검증을 원하는지 같은 선호는 설문 없이는 추론 불가 → 짧은 설문 필수.

---
description: Probe the project, survey the user, then seed personalized verification rules.
allowed-tools: [Read, Glob, mcp__agent-tracer__monitor_list_rules, mcp__agent-tracer__monitor_create_rule]
---

You are setting up Agent Tracer verification rules for the user's current project.
File detection alone is shallow — it tells us what is configured, not what the user
actually cares about. Combine filesystem probing with a short survey, then propose.

Step 1 — Probe filesystem (silent). Read candidates without showing raw output yet:
  - package.json → scripts.lint/test/typecheck/build
  - Cargo.toml, pyproject.toml (+ dev deps for ruff/black/pytest/mypy),
    go.mod, Makefile (glob targets)
  - tsconfig.json, eslint.config.*, biome.json, .prettierrc*
Build a detected set in memory.

Step 2 — Survey (conversational, <= 4 questions). Default-suggest from
detected signals so the user just confirms quickly. After Q1, always offer the
escape line: "Use detection only — skip survey".
  Q1) "What language(s) do you primarily work in on this project?"
       (suggest detected; user may correct, e.g., polyglot repo where only one lang matters)
  Q2) "Main framework(s)? (e.g., React, Nest, Django, Spring, none)"
  Q3) "When should I remind the agent to verify? Pick any:
        (a) after every edit  (b) before committing  (c) after refactors  (d) custom"
  Q4) "Any specific commands or patterns you ALWAYS want enforced?
       (free text, or 'skip')"

Step 3 — Synthesize. Combine detected + survey into 3~7 rule candidates.
Mapping examples:
  - "after every edit" + scripts.lint   → triggerPhrases=["edit","fix","implement","add"]
  - "before committing" + scripts.test  → triggerPhrases=["commit","PR","push"]
  - framework=React + scripts.test      → expectCommandMatches=["vitest","jest","playwright"]
  - language=Rust                       → expectCommandMatches=["cargo test","cargo clippy"]
  - Q4 free text                        → 1 dedicated rule per pattern
Encode survey context in each rule's rationale
(e.g. "TS + React, user picked: before commits").

Step 4 — Dedup + Confirm. Call monitor_list_rules({ scope: "global" })
to skip already-existing rules (signature-equivalent). Present remaining candidates
numbered, default-selected. User toggles.

Step 5 — Create. For each accepted, call monitor_create_rule with:
  scope="global", source="agent", severity="info",
  rationale (carries survey context for future audit).

Step 6 — Summarize. List created rule IDs and remind the user they can edit/delete
at http://127.0.0.1:5173/rules.

A-4. 왜 슬래시 명령 한 개로 끝나는가

  • 환경 감지: Claude 가 Read/Glob 으로 직접 — 별도 detect 모듈 불필요
  • 설문 인터랙션: 자연어 대화 — readline / inquirer 의존성 불필요
  • 인증: Claude Code 가 이미 인증 — API key 불필요
  • 적용 즉시성: monitor_create_rule 이 REST 와 동일 서비스로 들어가 실시간 enforcement 즉시 반영
  • 개인화: 파일 탐지(현 설정) + 설문(사용자 선호) 결합 — polyglot 레포·freeform 워크플로우도 적절히 매핑. 같은 React 프로젝트라도 "TDD 지향" vs "PR 직전 일괄 검증" 사용자에게 다른 rule 셋이 나옴
  • 지속성: 설문 답변은 각 rule 의 rationale 에 기록 → 다음 /setup-rules 호출 시 dedup 으로 중복 안 만들고, /rules UI 에서 "왜 이 rule 이 있는지" 추적 가능
설문 길이 트레이드오프
4문항이 처음엔 적정해 보여도 매번 호출하기엔 길 수 있음. 완화안: ① Q1 직후 "skip survey" 단축 경로, ② 두 번째 호출부터는 기존 rule 의 rationale 을 읽어 "이전 답변 유지하시겠어요?" 한 줄로 압축, ③ 검증 후 사용자가 답변 변경을 원하면 /setup-rules --resurvey 같은 인자로 전체 재실행.

2Phase B-1 · 웹 버튼 + 서버 SDK Agent

Phase B-1 대시보드에서 회고적으로 — 한 번의 클릭 · API key 필요 · 서버가 SDK Agent spawn

B1-1. UX 진입점

  • 위치: packages/web/src/features/inspector/tabs/rules/RulesTab.tsx 상단
  • 버튼: [ ✨ Generate rules from this task ]
  • 활성 조건: task.status === "completed" 이고 app_settings.anthropic.api_key 설정됨
  • 비활성 시 안내: "Configure API key in Settings to enable" / "Available after task completes"

B1-2. 백엔드 흐름

[1] user clicks Generate → POST /api/v1/tasks/:id/generate-rules
[2] server enqueue task_rule_generation_jobs(taskId, pending)
[3] RuleGenerationWorker.tick() picks up → load task + workspacePath + timeline summary
[4] server spawns Claude Agent SDK
    └ options: { cwd: task.workspacePath, allowedTools: ["Read","Glob","Grep"], maxTurns: 8 }
[5] SDK Agent explores workspace using its tools
[6] SDK Agent returns structured JSON rules → server validates (Zod)
[7] server bulk-creates via RuleCommandService.create() (source=agent, severity=info)
[8] EventBroadcaster emits rules.generated → web UI 토스트 + RulesTab refetch

B1-3. SDK Agent 의 시스템 프롬프트 (요지)

You are a verification-rule designer for Agent Tracer.

Goal: propose 3-5 rules that would catch whether a coding agent
properly completed a task similar to the one summarized below.

You may explore the codebase using Read, Glob, and Grep. Focus on
config files (package.json, tsconfig, eslint, biome, etc.), test
setup, and the files the task actually touched.

After exploration, output strict JSON only:
  { "rules": [{ name, expectTool, expectCommandMatches, severity, rationale }, ...] }

Constraints:
  - severity is always "info"
  - expectTool from {Bash, Read, Edit, Write, WebFetch, WebSearch}
  - rationale < 1 sentence
  - max 5 rules
  - do NOT propose rules that already exist (caller dedups by signature anyway, but try)

B1-4. 입력 컨텍스트 (caller → SDK Agent)

  • task.title + 첫 user.message body (head/tail trim < 2000자)
  • 도구별 사용 통계 (e.g., { Read: 8, Edit: 3, Write: 1, Bash: 12 })
  • 상위 5개 touched files (event_files 의 write_count desc)
  • 상위 10개 Bash 명령 (중복 제거 후)

워크스페이스 트리는 caller 가 미리 넣지 않음 — SDK Agent 가 필요 시 Glob 으로 직접 탐색하도록 위임. (그게 SDK Agent 도입 이유.)

B1-5. 인프라 추가 분량

  • 의존성: @anthropic-ai/claude-agent-sdk 1개 (server 패키지)
  • 새 모듈: packages/server/src/governance/rule/generation/ — rule.generation.service.ts, rule.generation.worker.ts, rule.generation.controller.ts
  • 새 테이블: task_rule_generation_jobs(id, task_id, status, attempts, error, started_at, completed_at)
  • WS 이벤트: rules.generated 한 종류
실패 모드
  • SDK Agent 가 JSON 외 텍스트를 뱉음 → Zod fail → 잡 status=failed, UI 토스트로 알림
  • cwd 가 더 이상 존재하지 않음 (workspacePath 가 지워진 프로젝트) → 즉시 실패, "workspace not found" 메시지
  • SDK 호출 timeout → 3회 backoff 재시도 후 실패
  • 같은 task 잡 중복 요청 → 진행 중 잡 ID 반환 (enqueue 거부)
  • API key 만료/오류 → 잡 실패, Settings 페이지로 유도하는 토스트

F리스크 / 열린 질문

리스크 1 · 슬래시 명령은 Claude Code 만 지원
Codex 쪽 사용자에게는 안 보임. 본 설계는 "우선 Claude Code 만 지원" 결정 하에 진행. Codex 지원이 필요해지면 그때 별도 진입점 (CLI prompt 부활 or Codex 의 자체 명령 시스템) 고려.
리스크 2 · MCP 툴이 잘못 호출되면 잘못된 rule 양산
완화: ① monitor_create_rule 서비스 단에서 Zod 스키마 검증, ② signature dedup 기존 활용, ③ 슬래시 명령 프롬프트에 "create rules ONLY after explicit user confirmation" 명시.
리스크 3 · SDK Agent 가 의도치 않은 파일 접근
완화: allowedTools: ["Read","Glob","Grep"] 만 — Edit/Write/Bash 차단. cwd 를 task.workspacePath 로 강제. maxTurns: 8 로 폭주 차단.
