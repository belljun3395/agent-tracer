Agent Tracer · Internal proposal

Hook 너머로 — Claude Code SDK를 활용한 모니터링 확장

현재 Agent Tracer는 hooks 한 가지 채널로만 데이터를 수집한다. 이미 모인 데이터(룰·이벤트·툴 사용·파일 흔적)는 풍부한데, Claude Code 플러그인/SDK가 제공하는 다른 표면(서브에이전트·슬래시 커맨드·스킬·MCP·Agent SDK headless)을 함께 쓰면 “모니터링”에서 “관여(intervene) + 해석(judge) + 재현(replay)”까지 단계를 끌어올릴 수 있다.

대상 저장소 agent-tracer 현재 브랜치 perf/display-title-cache 플러그인 v0.2.28 packages/runtime/.claude-plugin/plugin.json

1 현재 구성과 사용/미사용 표면

플러그인 디렉터리에는 plugin.json과 hooks/만 있다. commands/, agents/, skills/, 플러그인 번들 MCP는 모두 비어 있다. 즉, 데이터는 받지만 “에이전트에게 되돌려주는 통로”가 거의 없다.

지금 잘 되고 있는 것

  • 27/28 Claude hook 이벤트 수집 — action-only privacy contract
  • SQLite 기반 타임라인 · turn · rule_enforcement 영속화
  • Rule 시스템: trigger phrases → expected actions (command/file/web)
  • MCP stdio 서버에 약 20개 monitor_* 툴 등록
  • WebSocket 실시간 + OpenInference export

오직 “관찰”만 — 행동 영향력 0

  • Rule 위반이 감지돼도 에이전트가 모름: 대시보드에만 표시
  • PreToolUse 훅에 permissionDecision 응답 없음 → 차단 불가
  • Stop 훅의 decision:"block" 활용 없음 → “덜 끝낸 작업” 강제 재진입 불가
  • MCP 툴이 있지만 플러그인에 번들되지 않아서 별도 설치 필요
  • “왜 이 파일을 열었지?”에 답할 추론 레이어가 없음

2 재정의 — “모니터”에서 “Closed-loop 관찰자”로

현재 데이터 흐름: 에이전트 → 훅 → 서버 → 대시보드 → 사용자. 사용자만 시그널을 본다.
제안하는 흐름: 에이전트 ↔ 훅·MCP·서브에이전트 ↔ 서버(룰/판정) → (재투입) → 에이전트 + 사용자. 이미 잡고 있는 “파일/명령/검색” 시그널을 에이전트에게 되돌려주거나, 별도 판단자 Claude를 띄워서 “정말 지시한 일을 했는가?”에 구조화된 답을 만들어낸다.

3 추가 기능 제안 (Tier별)

각 Tier는 “모니터링의 위상”을 한 단계씩 올리는 묶음이다. 위쪽일수록 구현이 가볍고 가치 곡선이 가파르다.

TIER 1

Rule을 “관찰”에서 “관여”로 — 훅의 미사용 응답 활용

현재 훅은 한 방향. JSON 응답으로 차단·재진입·맥락 주입까지 가능.

1.1 Rule-as-Guardrail (PreToolUse 차단)

PreToolUse동기 훅
문제
Severity block 룰이 사실상 warn처럼 동작. 위반해도 도구는 실행됨.
변경
PreToolUse 훅이 서버 /api/v1/rules/match에 동기 조회 → 위반 시 permissionDecision: "deny" 반환.
가치
“커밋 메시지에 ‘fix’ 들어가면 typecheck부터”, “*.env 수정 금지” 같은 룰이 실제 제어가 된다. 대시보드는 “위반 시도” 로그로 격상.
// run-hook-claude.sh PreToolUse 응답
{
  "permissionDecision": "deny",
  "permissionDecisionReason": "Rule#42 violated: must run typecheck before Edit on packages/server",
  "additionalContext": "Active rule: . Run `npm run typecheck` first, then retry."
}

1.2 “덜 끝낸 작업” 재진입 (Stop 차단)

Stop
문제
사용자가 “테스트 추가해줘”라고 했는데 에이전트가 Write *.test.ts 없이 끝내도 시그널만 남음.
변경
Stop 훅이 현 turn의 expected-action 룰 통과 여부 확인 → 미충족 시 {decision:"block", reason}로 한 턴 더.
가치
기대-수행 갭을 턴이 끝나기 전에 닫는다. “말한 것을 진짜 했는가”에 대한 첫 자동 응답.

1.3 PreToolUse additionalContext로 룰을 에이전트에게 “속삭이기”

PreToolUse비차단
변경
차단까지 안 가도, 매 툴 호출 직전에 “현재 활성 룰 N개” 요약을 컨텍스트로 주입.
가치
에이전트가 스스로 룰을 인지하고 우회 시도/확인 멘트를 남긴다. 룰의 mental model이 사람·에이전트 양쪽에 공유됨.
TIER 2

플러그인을 “양방향”으로 — Commands / Subagent / MCP 번들

에이전트가 모니터에 직접 말 걸 수 있는 통로를 연다.

2.1 플러그인에 MCP 서버 번들

MCPplugin.json
현재
서버에 약 20개 monitor_* 툴 이미 존재. 그러나 사용자가 별도 claude mcp add 해야 함.
변경
plugin.json에 "mcpServers" 추가 — 플러그인 설치 한 번으로 훅 + MCP 동시 활성.
가치
설치 마찰 0. 에이전트가 monitor_thought, monitor_plan, monitor_question을 자연스럽게 호출 → 사고 흐름이 훅 데이터와 같은 타임라인에 합쳐짐.

2.2 슬래시 커맨드 묶음 /agent-tracer:*

commands/
예시
/agent-tracer:rules — 활성 룰을 컨텍스트로 주입
/agent-tracer:checkpoint <label> — 타임라인 마커
/agent-tracer:why <file> — “왜 이 파일을 열었지?” 추론 (Tier 3과 연결)
/agent-tracer:audit-me — 현재 턴 트레이스 자기 감사
/agent-tracer:rule "next time always…" — 자연어 → 룰 변환 후 POST
가치
사용자가 작업 중 한 줄로 모니터를 조작. 룰 등록 UX가 대시보드 클릭에서 CLI 한 줄로 단축.

2.3 auditor 서브에이전트

agents/
정의
agents/auditor.md — system prompt: “현 task의 timeline·rule을 읽고, expected action vs. observed tool 갭을 표 형태로 출력”.
트리거
메인 에이전트가 Agent 툴로 위임 → 독립 컨텍스트 윈도우에서 검증 → 결과를 turn에 첨부.
가치
메인 컨텍스트를 오염시키지 않으면서 “자기 검증” 단계를 강제. 사용자 개입 없이도 검증 결과가 timeline에 남음.
TIER 3

Agent SDK headless를 “판단자(Judge) 루프”로

현재 룰 매칭은 문자열 술어. SDK headless로 의미 기반 평가 추가.

3.1 룰 평가의 LLM 두 번째 티어

@anthropic-ai/claude-agent-sdkprompt caching
현재
RuleExpectation은 commandMatches: string[] / pattern: string의 정적 매칭.
변경
서버가 turn close 시 headless query()를 호출. 입력: 룰 NL 설명 + 그 턴의 timeline. 출력: {verdict, evidenceEventIds[], rationale}.
캐싱
시스템 프롬프트 + 룰 카탈로그에 cache_control 적용 → 턴마다 동일 토큰 prefix 재사용으로 비용 90%↓.
// packages/server/src/governance/verification/llm-judge.ts
const result = await query({
  model: "claude-haiku-4-5-20251001", // 저렴·빠른 모델로 충분
  systemPrompt: [
    { type: "text", text: JUDGE_SYS },
    { type: "text", text: rulesCatalog, cache_control: { type: "ephemeral" } }
  ],
  messages: [{ role: "user", content: renderTimeline(turn) }],
  outputSchema: VerdictSchema
});

3.2 이전 세션에서 룰 후보 자동 발굴

headlessbackground job
변경
N개 과거 task를 묶어 “이 프로젝트에서 에이전트가 암묵적으로 지키거나 자주 어긴 규칙은?”을 판단자 Claude에 질문 → 후보 룰로 UI 큐에 적재.
가치
“무엇을 검사해야 하는지조차 모르는” 초기 단계 사용자가 가장 답답해하는 부분을 자동화. 룰 시스템의 cold start 해결.

3.3 “Plan vs. Execution” 일치도 점수

ExitPlanModejudge
방법
ExitPlanMode 시 캡처한 plan 텍스트와 그 이후 실행된 tool sequence를 judge에 던져 “plan_adherence 0-1 + 누락/이탈 항목”.
가치
“어떻게 동작하는지 확인하고 싶다”는 사용자의 원 목적에 가장 정직한 지표. 룰 없이도 의미 있는 신호 생성.

3.4 트레이스 위에서 자연어 Q&A — /agent-tracer:ask

conversational
예시 질문
“같은 파일을 몇 번 다시 읽었어?”, “grep 패턴이 의미 있게 좁혀졌어?”, “/packages/server 외부를 만진 적 있나?”
구현
대시보드 Inspector에 chat 입력. 백엔드는 headless query()에 직렬화된 timeline JSON을 컨텍스트로 주입. 새 질문마다 같은 cache prefix 재사용.
TIER 4

재현 가능성 — Fork·Replay·Diff

“그때 다르게 했으면?”을 검증 가능한 실험으로.

4.1 Session Fork — 임의 timeline 지점에서 분기

resumeSDK 지원 확인
전제
훅이 모든 이벤트의 transcript_path를 보존. Agent SDK는 transcript 기반 resume을 지원.
기능
대시보드에서 임의 event ID를 “여기서 다시” → 새 task로 fork된 Claude 세션 자동 기동. 동일 MCP env + 다른 모델/룰로 재실행.

4.2 Behavioral Diff — Sonnet vs. Opus 같은 입력에 대해

headlessside-by-side
가치
사용자의 진짜 질문 — “이 에이전트가 잘 한 건가?” — 에 답하는 가장 직접적인 방법. tool 호출 sequence diff 시각화.
TIER 5

다른 SDK 사용자를 위한 “계측 라이브러리”

Claude Code 밖에서 SDK로 만든 에이전트도 같은 대시보드로 들어오게.

5.1 @monitor/sdk-tracer — Agent SDK wrapper

@anthropic-ai/claude-agent-sdk
현재
Agent Tracer는 Claude Code(plugin) + Codex(hook) 두 런타임만 자동 어댑터. Custom SDK 앱은 수동 HTTP 호출 필요.
변경
SDK query()를 감싸는 얇은 wrapper 패키지 — tool_use / tool_result 이벤트를 가로채 동일 /ingest/v1 페이로드로 전송.
import { tracedQuery } from "@monitor/sdk-tracer";

for await (const ev of tracedQuery({
  task: { title: "refactor authentication", source: "sdk-app" },
  prompt, model: "claude-opus-4-7"
})) { /* 평소처럼 사용 */ }
가치
대시보드 그대로 — 룰·Inspector·OpenInference export — 가 SDK로 만든 사내 봇에도 적용.

5.2 Rule Pack 번들

defaults
예시
“커밋 메시지에 fix/feat/chore prefix 강제”, “Read 없는 Edit 차단”, “.env 수정 시 deny”, “npm install 후 typecheck 필수”.
가치
설치 직후 “빈 룰 페이지” 문제 해소 (UI audit에서 발견됨). 한 명령으로 활성화 → 첫 인상이 “룰이 실제로 일하는 도구”로.
TIER 6

표면 정돈 — 작지만 효과 큰 SDK 활용

큰 구조 변경 없이 사용자 체감 가치 증가.

6.1 StatusLine에 “룰 컴플라이언스”

StatusLine
변경
기존 ctx N% · 5h N% · $X 옆에 rules 4/5 · 1⚠ 추가. 현 turn 룰 상태 1초마다 반영.
가치
대시보드 안 열고도 “지금 룰을 어기는 중”이 사용자 시야에 들어옴.

6.2 Skills로 “트레이스-aware 구현”

skills/
정의
skills/trace-aware/SKILL.md — “구현 전 active rule 조회 → 구현 후 monitor_verify 호출”을 표준화한 절차서.
가치
스킬 트리거 키워드(“implement”, “refactor” 등)에 자동 부착 → 에이전트가 모니터의 존재를 “원래 그런 것”처럼 흡수.

권장 진행 순서

  1. 2.1 MCP 번들 — 가장 싼 비용으로 가장 큰 “이미 만든 것의 활용도” 상승. 별도 설치 마찰 제거.
  2. 1.1 PreToolUse 차단 — 룰의 의미를 “관찰”에서 “관여”로 한 단계 격상. 데모 임팩트가 가장 큼.
  3. 3.1 LLM judge — 정적 매칭의 한계를 깬다. Haiku + prompt caching이면 운영비도 견딜 만함.
  4. 3.3 Plan adherence — “어떻게 동작하는지 확인”이라는 원 목적에 정확히 맞는 지표를 만들어낸다.
  5. 5.1 @monitor/sdk-tracer — 사용자 베이스를 Claude Code 외부로 확장. 장기 가치 가장 큼.

Tier 1·2는 기존 코드를 거의 안 건드린다 — packages/runtime에 디렉터리 추가와 hook 응답 JSON 한 줄 수준. Tier 3 이후는 서버에 새 모듈(governance/judge)이 생기지만, 기존 rule 모듈의 인터페이스를 그대로 만족시키므로 Inspector/Rules 탭 UI는 손대지 않아도 자연스럽게 연결된다.
