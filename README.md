# Agent Tracer

AI 코딩 에이전트의 실행 기록을 공통 이벤트 모델로 수집·시각화하고, 과거 기록에서 규칙과
반복 작업 절차를 추출해 다음 작업에 제공하는 시스템이다.

에이전트가 무엇을 읽고 고쳤으며 어떤 명령을 실행했는지는 세션이 끝나면 사라진다.
Agent Tracer는 이 기록을 이벤트 원장에 남기고 타임라인으로 보여준다. 반복되는 패턴에서 추출한
규칙과 반복 작업 절차는 다음 세션의 에이전트가 활용한다.

## 무엇을 하는가

**수집한다.** 에이전트 런타임에 플러그인으로 연결해 도구 호출, 파일 변경, 명령 실행, 사용자 입력,
권한 요청을 공통 이벤트 모델로 기록한다. 로컬 데몬이 이벤트를 모아 서버로 보내고, 서버가 응답하지
않으면 스풀에 쌓아 두었다가 복구된 뒤 보낸다. 수집 실패나 지연은 사용자의 작업을 막지 않는다.

**보여준다.** 대시보드가 작업과 세션과 턴을 타임라인으로 그린다. 어느 도구가 무엇을 했고
어디서 실패했는지 이벤트 단위로 파고들 수 있다.

**다음 작업에 제공한다.** AI 에이전트가 쌓인 실행 기록을 읽고 규칙과 반복 작업 절차인 레시피를
제안한다. 대화형 에이전트는 조회·변경·분석 기능을 공통 도구 계약으로 사용한다. 사용자가 승인한
규칙은 다음 세션부터 적용하고, 규칙·레시피·힌트를 에이전트 컨텍스트에 추가한다. 규칙을 이행하지
않은 턴은 종료 전에 다시 점검한다.

## 구조

쓰기와 조회가 데이터베이스를 공유하지 않는다.

에이전트가 보낸 이벤트는 쓰기 전용 API가 이벤트 원장에 추가만 한다. 변경 데이터 캡처가 원장의
변경을 스트림으로 전달하고, 조회 모델 생성기가 이를 소비해 조회 모델을 만든다. 조회 API는 조회
모델만 사용한다. 따라서 수집 경로는 조회 처리와 분리되고, 조회 모델은 언제든 이벤트 원장에서
다시 만들 수 있다.

AI 작업은 조회 API가 직접 실행하지 않는다. 내구성 있는 오케스트레이션 엔진이 워크플로를 돌리고,
재시도와 시간 초과와 취소를 소유한다.

## 배포 단위

| 패키지 | 하는 일 |
|---|---|
| `packages/kernel` | 네 배포 단위가 공유하는 어휘와 순수 도메인 규칙 |
| `packages/runtime` | 에이전트 쪽 수집기 플러그인과 로컬 데몬 |
| `packages/web` | 대시보드 |
| `packages/agents` | Python으로 구현한 에이전트 실행 백엔드 |
| `packages/server/apps/runtime-api` | 쓰기 전용 수집 API |
| `packages/server/apps/tracer-api` | 조회와 AI 작업 등록과 실시간 전송 |
| `packages/server/apps/projector` | 이벤트 원장을 소비해 조회 모델을 생성 |
| `packages/server/apps/ai-agent-worker` | AI 작업의 오케스트레이션 |

각 배포 단위는 서로를 직접 import하지 않는다. 공유 커널로만 연결된다.

## 쓰기 시작하기

이 시스템은 두 조각이다. 실행 기록을 수집하는 에이전트 쪽 플러그인과, 그것을 받아 저장하고 보여주고
분석하는 서버다. 둘 다 있어야 쓸 수 있다. 플러그인만 깔면 이벤트가 로컬 스풀에 쌓이다 버려지고,
서버만 띄우면 아무것도 들어오지 않는다.

Node 24와 도커가 필요하다. 다른 Node 버전에서는 네이티브 의존성이 로드되지 않고 훅 번들이
실행되지 않는다.

먼저 서버를 띄운다. 앱 이미지를 이 저장소에서 빌드하므로 클론이 필요하다.

```bash
git clone https://github.com/belljun3395/agent-tracer.git
cd agent-tracer
npm run stack:up     # 배포되는 이미지 그대로 인프라와 앱을 전부 띄운다
```

수집 진입점이 `http://127.0.0.1:3847`에, 대시보드가 `http://127.0.0.1:5173`에 열린다.
내릴 때는 `npm run stack:down`이다.

그 다음 감시할 에이전트에 플러그인을 붙인다. Claude Code에서는 이렇게 한다.

```
/plugin marketplace add belljun3395/agent-tracer
/plugin install agent-tracer-monitor@agent-tracer
```

플러그인은 기본으로 `http://127.0.0.1:3847`을 향한다. 서버를 다른 호스트에 띄웠다면
`MONITOR_BASE_URL`로 옮긴다. 훅이 자기 버전과 데몬의 버전을 비교해 다르면 데몬을 갱신하므로,
플러그인을 업데이트한 뒤 수동으로 재시작할 필요가 없다.

이제 에이전트를 평소처럼 쓰면 된다. 대시보드에 작업과 세션이 쌓이고, 실행 기록이 모이면
규칙과 레시피를 제안받을 수 있다. 사용자가 승인한 규칙은 다음 세션부터 적용된다.

## 개발하기

소스에서 직접 띄운다.

```bash
npm ci
npm run infra:up     # 데이터베이스, 브로커, 변경 데이터 캡처, 검색, 오케스트레이션 엔진
npm run dev          # 마이그레이션을 먼저 돌리고 전 서비스를 띄운다
```

`npm run dev`는 `packages/agents`를 `uv`로 함께 띄우므로 Node 24와 도커 말고 uv와 Python 3.12도
있어야 한다. 셋을 띄우지 않고 서버만 볼 것이면 `npm run stack:up`을 쓴다.

플러그인을 마켓플레이스가 아니라 이 작업 트리에서 물리려면 감시할 프로젝트에
`npm run setup:external -- --target <path>`를 돌린다.

데몬은 자체 제어 화면을 로컬에서 제공한다. 파이프라인 상태, 개입 기록, 규칙 발동, 스풀,
데드레터 재투입을 여기서 본다. 대시보드가 아니라 데몬이 제공하는 이유는 파이프라인이 죽었을 때
쓰는 화면이기 때문이다.

## 검증

```bash
npm run lint         # 형식, 타입, 미사용 코드, 구조 허용치
npm run test         # 단위 테스트
npm run lint:deps    # 의존 그래프 규칙
```

셋이 통과해야 작업이 끝난 것이다. 단 이 셋은 TS 워크스페이스만 덮는다. `packages/agents`는
npm 워크스페이스 밖이라 여기에 걸리지 않으므로, 그쪽을 고쳤다면 `packages/agents/README.md`의
검증 루프(ruff·mypy·pytest·주석·의존 검사)를 따로 돌려야 한다.

구조 규칙은 `ARCHITECTURE.md`가, 결정의 근거는 `docs/adr/`가 소유한다.
