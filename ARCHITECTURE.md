# 구조 규칙

코딩 에이전트 런타임의 활동을 수집하고 조회하고 분석하는 모니터링 시스템이다.
쓰기와 읽기가 데이터베이스를 공유하지 않고 변경 데이터 캡처로 연결된다.

이 문서는 어디에 무엇을 쓸지 판단하는 데 필요한 전부를 담는다. 다른 문서를 읽지 않아도 된다.

## 1. 세 개의 축

디렉터리는 세 축으로만 나뉘고, 순서가 고정되어 있다.

```
배포 단위  →  도메인  →  계층
```

1축은 배포 단위다. 따로 배포되고 따로 죽는 것이 따로 있는다. `packages/*`가 곧 배포 단위이며
언어는 축이 아니다.

2축은 도메인이다. 배포 단위의 `src/domain/` 아래에 도메인 슬라이스가 온다.

3축이 비로소 계층이다. 계층 이름은 도메인 슬라이스 **안쪽에서만** 등장한다.
`src/` 바로 아래에 `application/`이나 `controller/`가 보이면 축이 뒤집힌 것이다.

한 디렉터리 레벨은 한 축으로만 나눈다. 같은 폴더에 도메인 이름과 기술 이름이 나란히 보이면
그것이 잘못된 자리다.

## 2. 배포 단위의 뼈대

모든 배포 단위가 같은 뼈대를 갖는다.

```
src/
  domain/
    <slice>/
      inbound/       구동 어댑터. 바깥에서 들어오는 요청을 응용 계층으로 넘긴다
      application/   유스케이스. 이 도메인이 하는 일
      port/          아웃바운드 계약. 이 도메인이 바깥에 요구하는 것
      adapter/       포트의 구현. 기술이 여기 있다
      model/         순수 도메인. 프레임워크도 IO도 모른다
  config/            앱 전역 기술 기반
  support/           순수 유틸
  main.ts            조립 근원
```

`__fakes__/`는 `port/` 안에 산다. 포트의 인메모리 대역이므로 포트와 함께 움직인다.

## 3. 계층 어휘는 다섯이고 기술은 접미사가 말한다

역할이 같으면 디렉터리 이름이 같다. 그래야 규칙 하나가 전부를 덮는다.
기술의 차이는 디렉터리가 아니라 파일 접미사가 말한다.

| 계층 | HTTP API | 메시지 소비 | 잡 오케스트레이션 | 에이전트 훅 |
|---|---|---|---|---|
| `inbound/` | `*.controller.ts` | `*.consumer.ts` | `*.workflow.ts` · `*.activity.ts` | `*.hook.ts` |
| `application/` | `*.usecase.ts` | `*.usecase.ts` · `*.projection.ts` | `*.usecase.ts` | `*.usecase.ts` |
| `port/` | `*.port.ts` | `*.port.ts` | `*.port.ts` | `*.port.ts` |
| `adapter/` | `*.adapter.ts` | `*.adapter.ts` | `*.adapter.ts` | `*.adapter.ts` |
| `model/` | `*.model.ts` · `*.policy.ts` · `*.spec.ts` | 〃 | 〃 | 〃 |

`*.usecase.ts`는 응용 계층의 진입점이다. 하나의 공개 실행 메서드를 갖고 다른 유스케이스를
호출하지 않는다.

`*.projection.ts`는 진입점이 아니라 유스케이스가 밟는 단계다. 유스케이스가 호출할 수 있고,
`inbound/`는 이를 직접 참조하지 못한다. 원장 배치처럼 한 트랜잭션 안에서 여러 갈래로
갈라지는 처리에만 쓴다.

## 4. 의존 방향

슬라이스 안쪽 규칙이다.

```
inbound      →  application, model, support
application  →  port, model, support
port         →  model
adapter      →  port, model, config, support, 외부 SDK
model        →  support
```

`application`은 `adapter`를 모른다. `model`과 `port`는 프레임워크를 모른다.
`adapter`는 `application`과 `inbound`를 모른다.

슬라이스 사이 규칙이다.

```
슬라이스  ↛  형제 슬라이스
```

배포 단위 전역 규칙이다.

```
config   ↛  domain
support  ↛  config, domain
조립 근원(main.ts, *.module.ts)만 모든 슬라이스를 안다
```

배포 단위 사이 규칙이다.

```
server ⊥ web ⊥ runtime ⊥ agents      직접 import 금지. 예외 없음
넷 다 kernel만 공유한다
kernel은 어느 배포 단위도 알지 않는다
```

## 5. config와 support

`config/`는 앱 전역의 기술 기반이다. 데이터베이스 연결, 메시지 브로커 배선, 인증 가드,
예외 필터, 응답 봉투, 헬스 서버, 로깅, 메트릭, 언어 모델 공급자 실행기가 여기 있다.
IO와 배선이 있고 도메인 어휘가 없다.

`support/`는 순수 유틸이다. 상태도 IO도 도메인 어휘도 없다. 입력을 받아 출력을 낸다.

셋 중 하나를 고르는 판단은 이렇다. 도메인 어휘가 있으면 `domain/`이다. 없고 IO나 배선이
있으면 `config/`다. 없고 순수하면 `support/`다.

`config/`와 `support/` 아래에는 `*.usecase.ts`와 `*.model.ts`가 존재할 수 없다.

슬라이스의 `adapter/`와 앱의 `config/`는 이렇게 갈린다. 슬라이스가 소유한 포트를 구현하면
`adapter/`이고, 도메인 포트 없이 기술 자체를 제공하면 `config/`다. 데이터베이스 커넥션은
`config/`이고 그 커넥션으로 태스크를 읽는 저장소는 `domain/task/adapter/`다.

## 6. 중복

**도메인 내부의 중복은 금지한다.** 한 슬라이스 안에서 여러 유스케이스가 같은 코드를 쓰면
그 슬라이스의 `model/`로 내리거나 유스케이스를 합친다.

**도메인 간의 중복은 허용한다.** 두 슬라이스가 비슷한 코드를 갖는 것이 기본값이다.
공통 모듈로 끌어올려 두 도메인이 한 코드를 함께 쓰게 만드는 것이 오히려 금지다.
에이전트 워커에서 레시피 도메인과 정리 도메인이 각자의 이벤트 조회 도구를 갖는 것이 정상이고,
이를 하나의 공용 도구로 묶으면 한쪽의 요구가 다른 쪽을 흔든다.

도메인이 슬라이스로 갈라져 있다는 것은 그 도메인들이 따로 변한다는 선언이다.
그 선언을 지키는 값이 중복의 비용보다 크다.

이 규칙은 기계가 잡지 못한다. 커밋 리뷰가 소유한다.

## 7. 포트

아웃바운드 의존은 예외 없이 포트 뒤에 둔다. 데이터베이스, 검색, 메시징, 언어 모델, HTTP,
파일시스템, 시계, 난수까지 전부 포함한다.

포트는 슬라이스가 소유한다. 슬라이스마다 `port/` 하나가 그 슬라이스의 모든 아웃바운드 계약을
담는다. 포트는 "이 도메인이 바깥에 무엇을 요구하는가"의 선언이므로 도메인과 함께 움직인다.

모든 포트를 모으는 전역 포트 패키지는 만들지 않는다. 도메인이 사라져도 남는 잡동사니가 되고,
슬라이스 독립 규칙을 무력화한다.

포트의 구현이 이미 있으면 어댑터를 새로 쓰지 않는다. 읽기 모델의 저장소처럼 공유 라이브러리가
포트를 그대로 만족하는 경우, 슬라이스의 조립 근원이 그 구현을 포트에 바인딩한다. 위임만 하는
어댑터를 한 겹 두는 것은 이름만 늘리고 아무 경계도 세우지 않는다. 어댑터는 포트와 기술 사이에
번역이 필요할 때 쓴다.

## 8. 배포 단위

### kernel

네 배포 단위가 공유하는 최내곽 패키지. 프레임워크 무의존.
경계를 넘는 값의 어휘와 스키마뿐 아니라 규칙 판정과 레시피 매칭 같은 순수 도메인 규칙도
소유한다. 로컬 플러그인이 오프라인에서 서버와 같은 판정을 내야 하므로 공유가 필연이다.

zod는 파일명이 `*.schema.ts`인 모듈에만 존재한다. runtime과 web은 이 스키마를 값으로
import하지 못한다. 훅 번들과 웹 번들이 자립해야 하기 때문이다. 타입으로만 쓴다.

### server/apps/runtime-api

쓰기 전용. 에이전트가 보내는 이벤트를 원장에 적재한다. 읽기 모델을 알지 못한다.

도메인은 `ingest`(수집), `cold`(원장 보관과 보존), `health`다.

### server/apps/tracer-api

읽기와 잡과 실시간 전송.

도메인은 `task`, `timeline`, `rule`, `recipe`, `job`, `search`, `settings`, `user`,
`session`, `cleanup`, `health`다.

### server/apps/projector

원장을 소비해 읽기 모델로 투영한다. **이 배포 단위만 슬라이스의 축이 도메인이 아니라 기능이다.**

`src/domain/` 래퍼는 그대로 쓴다. 래퍼는 "여기부터 비즈니스 슬라이스"라는 표시이고, 축은
슬라이스의 이름이 말한다. 조회 API의 슬라이스가 도메인 명사인 것과 달리 여기서는 기능 동사다.
래퍼가 같으므로 규칙도 한 벌로 두 배포 단위를 덮는다.

```
src/
  domain/
    project/    원장을 읽기 모델로 투영한다
    notify/     투영 결과를 알림으로 발행한다
    index/      검색 인덱스를 채운다
    export/     관측 파이프라인으로 내보낸다
    recover/    중단된 잡과 태스크와 스텝을 회수한다
  config/  support/  main.ts
```

`project/`의 진입점은 유스케이스 하나다. 배치를 받아 트랜잭션을 열고, 레코드마다 해당 투영
단계를 호출하고, 마지막으로 적용한 시퀀스를 갱신하고 커밋한다. 개별 투영은 `*.projection.ts`이며
자기가 필요한 저장소 포트만 받는다.

여기에 도메인 축을 씌우면 타임라인과 실행과 레시피와 규칙이 형제 슬라이스가 되어 서로 부르지
못하고, 그 금지를 우회할 레지스트리 장치를 만들어야 한다. 트랜잭션 경계가 배치 하나이므로
이 투영들은 원래 한 몸이다. 축이 그 사실을 말해야 한다.

기능 축이라도 계층 어휘는 같다. 다섯 계층이 그대로 적용된다.

### server/apps/ai-agent-worker

AI 잡의 오케스트레이션. 도메인은 `recipe`, `title`, `cleanup`이다.

각 슬라이스는 이렇게 생겼다.

```
recipe/
  inbound/     recipe.workflow.ts   recipe.activity.ts
  application/ scan.recipe.usecase.ts
  port/        recipe.agent.port.ts   recipe.repository.port.ts
  adapter/     recipe.sdk.agent.adapter.ts   recipe.graph.agent.adapter.ts
  model/       recipe.spec.ts   task.summary.model.ts
```

SDK 백엔드의 프롬프트와 출력 스키마와 도구 명세는 제품 지식이므로 `model/`에 있다. 파이썬
백엔드는 자기 명세를 자기 언어로 갖는다. 두 명세가 같아야 하는 사실은 커널의 계약 픽스처가
소유하고 두 언어의 테스트가 그 파일 하나를 읽는다.

`*.workflow.ts`는 결정적이어야 한다. `*.activity.ts`, `adapter/`, `port/`, `config/`,
Node API를 import하지 못한다. 이것은 오케스트레이션 엔진의 외부 제약이다.

언어 모델 공급자 실행기는 `config/llm/`에 있다. 도메인 어휘를 모르고, 프롬프트를 넣으면
구조화된 출력을 낸다. 슬라이스의 어댑터가 이 실행기와 자기 명세를 조합한다.

### server/libs/platform

여러 앱이 함께 쓰는 기술 기반. 설정 로딩, 데이터베이스 팩토리, 브로커 팩토리, 인증 토큰,
시계, 식별자 생성.

### server/libs/tracer-domain

읽기 모델의 엔티티와 저장소 구현, 그리고 그 모델의 도메인 규칙. projector가 쓰고 tracer-api가 읽는다.
스키마와 마이그레이션의 소유자가 하나여야 하므로 라이브러리로 둔다.

투영기와 조회 API가 같은 판정을 내려야 하는 규칙은 여기 있다. 태스크 상태 전이, 턴 조립,
이벤트 표현, 레시피 수명주기가 그렇다. 앱은 서로를 import하지 못하므로 이 규칙이 앱으로
내려가면 두 벌이 되고, 두 벌이 갈라지면 쓴 모델과 읽은 모델이 어긋난다.

스키마는 엔티티가 단독으로 소유한다. 마이그레이션은 엔티티에서 생성하며, 손으로 쓴 DDL로
컬럼이나 인덱스를 더하지 않는다. 기본값과 부분 인덱스까지 엔티티가 선언해야 새 환경의 스키마가
기존 환경과 같아진다.

### packages/agents

Python으로 구현한 에이전트 실행 백엔드. npm workspace가 아니라 자체 검증 루프를 쓴다.
저장소를 갖지 않지만 무상태는 아니다. 자기 프롬프트와 도구와 검증 정책을 소유하며, 워커와
같아야 하는 사실은 커널의 계약 픽스처가 강제한다. 디렉터리 이름은 파이썬의 관례를 따르며
통일 대상이 아니다.

### packages/runtime

에이전트 쪽 수집기 플러그인. 도메인은 `ingest`, `guardrail`, `recipe`, `hint`, `rulegen`,
`binding`, `session`, `turn`이다.

에이전트 런타임 어댑터는 `agent/` 아래에 있고 와이어 포맷만 안다.

```
src/
  domain/     ingest/ guardrail/ recipe/ hint/ rulegen/ binding/ session/ turn/
  agent/
    claude-code/    훅 엔트리와 페이로드 리더
  daemon/     장수 프로세스. 조립 근원과 제어 화면
  config/  support/
```

도메인은 `agent/`와 `daemon/`을 모른다. 어댑터는 도메인의 응용 계층만 부른다.

어댑터가 지금 하나뿐이어도 `agent/` 경계는 유지한다. 두 번째 런타임이 생기는 순간 이 경계가
없으면 와이어 파싱과 제품 규칙이 다시 섞인다.

이 축이 중요한 이유가 있다. 도구 호출을 어떤 이벤트로 만들 것인가는 제품 규칙이지 어댑터의
사정이 아니다. 절단 상한, 도구 응답 캡처, 파일 타깃 수집, 제목과 본문 파생이 전부
`domain/ingest/model/`에 단 하나 있고 어댑터는 와이어 포맷을 읽기만 한다. 어댑터가 조형을
조금씩 다르게 하면 같은 도구 호출이 유입 경로에 따라 다른 이벤트가 되고, 그 차이는 원장에
영구히 남는다. 컨텍스트 주입 텍스트도, 규칙 생성 프롬프트 명세도 같은 이유로 도메인이
단독으로 갖는다.

데몬은 배포되는 장수 프로세스다. 훅과 데몬 사이의 소켓 계약은 `daemon/port/`가 단독으로
소유하고 클라이언트와 서버가 같은 타입을 쓴다.

### packages/web

대시보드. 여섯 레이어로 나눈다.

```
src/
  app/        조립, 라우팅, 프로바이더, 전역 스타일
  pages/      라우트 단위 화면
  widgets/    독립 UI 블록
  features/   사용자 행동 단위
  entities/   도메인 모델과 그 표현
  shared/     UI 킷, 유틸, API 클라이언트, 설정
```

규칙은 둘이다. 상위 레이어만 하위 레이어를 import한다. 같은 레이어의 슬라이스끼리 직접
import하지 않는다.

운영 UI 문자열은 영어로 쓴다. 한국어 설명 문구는 단일 카탈로그가 소유하고, 화면은 그 카탈로그의
메시지를 렌더링한다.

## 9. 규칙은 파일 이름을 모른다

아키텍처 규칙이 특정 파일 이름을 알아야 한다면 그 파일이 잘못된 자리에 있는 것이다.
규칙은 디렉터리 역할과 파일 접미사만으로 표현된다.

규칙의 종류는 셋뿐이다.

**구조 규칙.** 계층 방향, 슬라이스 독립, config와 support의 무지, 배포 단위 격리.

**접미사 규칙.** `*.usecase.ts`는 `*.usecase.ts`를 부르지 않는다.
`*.query.controller.ts`는 명령 유스케이스를 부르지 않는다.
`*.workflow.ts`는 활동 구현과 어댑터를 모른다.
`inbound/`는 `*.projection.ts`를 모른다.

**기술 봉인 규칙.** ORM은 `adapter/`와 `config/`와 마이그레이션 밖으로 새지 않는다.
프레임워크는 `model/`과 `port/`에 없다. zod는 `*.schema.ts`에만 있다.

`architecture.manifest.mjs` 하나가 이 규칙들의 유일한 서술이다. 의존 그래프 검사기와 린터가
이 선언을 읽어 규칙을 계산하고, 경로 별칭 설정이 이 선언에서 생성된다. 같은 경계를 두 곳에
손으로 적지 않는다.

새 규칙을 하나 더 추가하고 싶어지면, 그것은 구조를 고치라는 신호다.

## 10. 주석과 테스트

주석에 남기는 것은 셋뿐이다.

- 포트의 계약. 여러 구현체가 따라야 하는 역할. 포트마다 한 문장.
- 경계를 넘는 값의 외부 의미. 요청과 응답의 필드가 무엇을 뜻하는지.
- 코드로 표현할 수 없는 외부 제약. 프로토콜, 단위, 플랫폼 한계.

테스트가 소유하는 것은 이렇다. 분기, 필터, 반복, 재시도, 실패 처리, 경계값, 회귀 사례,
내부 데이터 변환, 실행 순서.

주석은 한글 한 문장으로 쓴다. 코드 식별자와 기술 고유명사는 영어로 쓸 수 있다.
TypeScript는 JSDoc, Python은 docstring을 쓴다. 구현 안의 줄 주석은 그 줄을 이해하는 데
외부 제약이 꼭 필요할 때만 쓴다.

**한 선언에 주석은 한 벌이다.** 줄 주석과 JSDoc을 나란히 달지 않는다. 둘을 함께 달면 어느 쪽이
계약이고 어느 쪽이 사정인지 흐려진다. 외부 제약이 계약의 일부라면 그 한 문장 안에 넣고,
아니라면 지운다. 린터가 이를 거부한다.

주석과 문서는 자기완결이다. 외부 링크, 다른 문서 참조, 결정 문서 번호 인용, 이력 서술을
달지 않는다. 그 문장만 읽고 이해되어야 한다.

테스트 이름은 한글 행위 문장이다. `describe`는 대상을, `it`은 "…한다"를 쓴다.

계층마다 테스트의 성격이 다르다.

| 계층 | 무엇을 검증하는가 | 대역 |
|---|---|---|
| `model/` | 순수 규칙. 입력이 어떤 판정을 내는가 | 없음 |
| `application/` | 유스케이스의 행위. 어떤 포트를 어떻게 부르고 무엇을 반환하는가 | `port/__fakes__/` |
| `adapter/` | 포트 계약의 준수 | 실제 기술 또는 모의 전송 |
| `inbound/` | 와이어 파싱과 위임 | 유스케이스 모의 |

유스케이스는 예외 없이 인접 테스트를 갖는다.

## 11. 품질 기준

300줄을 넘는 소스 파일을 만들지 않는다.
테스트 없는 유스케이스를 만들지 않는다.
주석 언어 규칙 위반을 남기지 않는다.

이 셋의 허용치는 0이며 올리지 않는다. 예산을 두고 백로그를 관리하지 않는다.
지킬 수 없는 파일이 나오면 그 파일이 여러 책임을 지고 있다는 뜻이므로 나눈다.
크기만으로 나누지는 않는다. 단일 책임인데 긴 파일은 책임이 하나임을 먼저 확인한다.

## 12. 호환

저장 포맷이나 계약이 바뀌면 즉시 컷오버한다. 구버전을 감지해 통과시키는 코드를 남기지 않고,
쌓인 데이터를 보존하기 위한 마이그레이션도 만들지 않는다. 소비자의 버전을 함께 올린다.

## 13. 커밋

커밋 하나는 수직 슬라이스 하나다. 도메인 슬라이스 하나가 인바운드부터 모델까지, 그 슬라이스의
테스트까지 함께 들어온다. 커밋 하나를 체크아웃하면 검증이 전부 통과하고, 그 슬라이스가 무엇을
하는지 변경분만 읽고 알 수 있다.

계층별로 커밋을 나누지 않는다. 포트만 있고 어댑터가 없는 상태는 확인할 수 있는 단위가 아니다.

메시지는 `타입(범위): 한글 제목` 형식이다. 타입은 feat, fix, refactor, test, docs, chore, ci다.
제목은 제목만 읽고 무엇을 했는지 아는 행위 문장이고 60자를 넘지 않는다.
본문은 없거나 네 줄 이내다. 정형 블록을 쓰지 않는다.

허용되지 않은 공동 작성자 트레일러를 커밋에 남기지 않는다. 커밋 훅이 이를 거부한다.
