# Agent Tracer Plugin ‚Äî Data Flow

This document describes how each hook handler processes data: what fields come from
Claude Code on stdin, what the plugin generates itself, and what gets sent to the
Agent Tracer monitor API.

---

Each hook process is short-lived. As of plugin v0.2.0 the plugin no longer
maintains per-session FS caches ‚Äî every handler re-calls
`/api/runtime-session-ensure` (idempotent on the server). The only plugin-local
state is the transcript cursor, which tracks byte offsets into Claude Code's
append-only transcript JSONL so Stop/SubagentStop can tail only new entries.

> **Migration note:** Plugin versions < 0.2.0 wrote to
> `.claude/.session-cache/` and `~/.claude/.session-history.json`. As of
> v0.2.0 those files are no longer read or written. Safe to delete any
> existing copies. The transcript cursor moved from
> `.claude/.session-cache/<sid>-transcript-cursor.json` to
> `.claude/.transcript-cursors/<sid>.json`; the plugin reads the legacy
> path once on upgrade and drains it on the first successful new-location write.

## Data origin legend

| Symbol | Meaning |
|--------|---------|
| **[CC]** | Value comes directly from Claude Code's hook payload (stdin) |
| **[GEN]** | Value is generated by the plugin at runtime |
| **[ENSURE]** | Value is returned from `/api/runtime-session-ensure` (re-called each hook; server is idempotent) |
| **[ENV]** | Value comes from an environment variable |
| **[TX]** | Value is parsed from the Claude Code session transcript JSONL (`transcript_path`) |

## Semantic metadata

The plugin sends **raw payloads only** ‚Äî it does not compute `subtypeKey`,
`toolFamily`, `lane`, or any other semantic classification. All semantic
derivation happens server-side in `@monitor/classification` via
`classifyEvent()` + `deriveSemanticMetadata()` during event ingestion.

The only metadata the plugin attaches is handler-specific context required
to reconstruct the event on the server (tool inputs, file paths, mcp server/tool,
tool-use IDs, etc.). Each section below lists the exact fields each handler emits.

## Session lifecycle

### 1. SessionStart ‚Üí `SessionStart.ts`

**Claude Code provides [CC]:**
```
session_id        ‚Äî unique session identifier
source            ‚Äî "startup" | "resume" | "clear" | "compact"
cwd               ‚Äî current working directory
model             ‚Äî model in use (e.g. "claude-sonnet-4-6")
hook_event_name   ‚Äî "SessionStart"
```

**Plugin generates [GEN]:**
```
title   ‚Äî derived from source: "Session started", "Session resumed", etc.
body    ‚Äî derived from source: human-readable description
trigger ‚Äî same as source, stored as metadata
```

**Plugin writes to disk:** none. The server records the start timestamp from
the `/api/runtime-session-ensure` call; the plugin no longer persists session
metadata locally.

**API calls:**
```
POST /api/runtime-session-ensure
  runtimeSource     [GEN] "claude-plugin"
  runtimeSessionId  [CC]  session_id
  title             [GEN] defaultTaskTitle() = "Claude Code ‚Äî <basename>"
  workspacePath     [ENV] $CLAUDE_PROJECT_DIR

POST /ingest/v1/events  { kind: "context.saved", ‚Ä¶ }
  taskId    [ENSURE] from ensureRuntimeSession response
  sessionId [ENSURE] from ensureRuntimeSession response
  title     [GEN]   "Session started" / "Session resumed" / ‚Ä¶
  body      [GEN]   description text
  lane      [GEN]   "planning"
  metadata:
    trigger [CC]    source ("startup" | "resume" | "clear" | "compact")
```

---

### 2. SessionEnd ‚Üí `SessionEnd.ts`

**Claude Code provides [CC]:**
```
session_id   ‚Äî same session_id from SessionStart
reason       ‚Äî "prompt_input_exit" | "resume" | "logout" | "clear" | ‚Ä¶
cwd          ‚Äî current working directory
```

**Plugin generates [GEN]:**
```
completionReason  ‚Äî "explicit_exit" (prompt_input_exit) or "runtime_terminated"
endedAt           ‚Äî Date.now() at time of hook (included in session.ended metadata)
```

**Plugin writes to disk:** none. The transcript cursor for this session is
deleted (see cleanup below) but no session records are persisted locally.

**Plugin cleans up:**
```
.claude/.transcript-cursors/<sessionId>.json  (deleted)
```

**API calls:**
```
POST /api/runtime-session-ensure    # re-ensure to resolve (taskId, sessionId)
  runtimeSource     [GEN]   "claude-plugin"
  runtimeSessionId  [CC]    session_id
  title             [GEN]   defaultTaskTitle()
  workspacePath     [ENV]   $CLAUDE_PROJECT_DIR

POST /api/runtime-session-end
  runtimeSource     [GEN]   "claude-plugin"
  runtimeSessionId  [CC]    session_id
  summary           [GEN]   "Claude Code session ended (<reason>)"
  completionReason  [GEN]   "explicit_exit" | "runtime_terminated"
  (no metadata object)

POST /ingest/v1/events  { kind: "session.ended", ‚Ä¶ }
  taskId    [ENSURE] from runtime-session-ensure
  sessionId [ENSURE] from runtime-session-ensure
  title     [GEN]   "Session ended (<reason-description>)"
  body      [GEN]   "Claude Code session ended (<reason>)."
  lane      [GEN]   "user"
  metadata:
    reason            [CC]   payload.reason
    completionReason  [GEN]  mapped from reason
    source            [GEN]  "session-end"
    sessionEndedAt    [GEN]  ISO timestamp of endedAt
    transcriptPath    [CC]   payload.transcript_path (omitted if absent)
    permissionMode    [CC]   payload.permission_mode (omitted if absent)
    cwd               [CC]   payload.cwd             (omitted if absent)
```

> `clear` reason is skipped ‚Äî SessionStart handles the new session that immediately
> follows a /clear command.

---

### 3. Session resume flow

Session history is owned by the server as of plugin v0.2.0. The plugin no
longer writes `~/.claude/.session-history.json`; `runtime-session-ensure` is
idempotent on `runtimeSessionId`, so reattaching to an existing task is a
matter of calling ensure again with the original session ID.

External tools that need to enumerate past sessions query the server's
lifecycle events (`session.started` / `session.ended`) instead of reading a
local history file.

**Claude Code provides [CC]:**

## Subagent session linking

The plugin tracks subagent sessions with a virtual runtime session ID:
`sub--<agent_id>`.

```
SubagentStart fires ‚Üí SubagentStart.ts eagerly resolves:
  parent session_id                [CC]
  child runtimeSessionId           [GEN] "sub--<agent_id>"

  POST /api/async-task  (asyncStatus: "running")
    metadata:
      agentId      [CC]    agent_id
      agentType    [CC]    agent_type
      parentTaskId [ENSURE] parent task ID
      childTaskId  [ENSURE] child task ID

Next PreToolUse / PostToolUse inside the subagent fires:
  resolveEventSessionIds(...)
  ‚Üí resolveSubagentSessionIds(...)
  ‚Üí POST /api/runtime-session-ensure (idempotent ‚Äî server returns same child IDs)

SubagentStop fires:
  POST /api/async-task  (asyncStatus: "completed")
    metadata:
      agentId      [CC]    agent_id
      agentType    [CC]    agent_type
      parentTaskId [ENSURE] parent task ID
```

This eager child-session creation keeps the subagent's task linked to the parent
before any tool events from the child are posted, without needing a separate
registry file.

## 14. Transcript tail lifecycle

Hook payloads carry only shallow `tool_input` / `tool_response` fields. They
cannot see the model's `thinking` blocks, intermediate narration text between
tool calls, or the system-injected attachments that Claude Code threads into
context (task reminders, plan mode, skill listings, tool/MCP deltas, nested
memory). All of that does appear in the session transcript JSONL that Claude
Code writes at `transcript_path`. Stop and SubagentStop tail this file on each
turn boundary and publish the new entries as events.

**Trigger points:**
- `Stop.ts` tails the parent session transcript (`transcript_path`).
- `SubagentStop.ts` tails the child session transcript (`agent_transcript_path`)
  and publishes events onto the child task via `resolveEventSessionIds`.

**Cursor file (per logical session):**
```
<PROJECT_DIR>/.claude/.transcript-cursors/<sessionId>.json
  { lastEmittedUuid: string | null, byteOffset: number, fileSize: number }
```
(Legacy location `.session-cache/<sessionId>-transcript-cursor.json` is read
as a one-time migration fallback and deleted after the first new-location write.)
- Parent sessions use the real session_id.
- Subagents use the virtual ID `sub--<agent_id>` so sibling agents stay
  isolated and cursors are deleted with the agent lifecycle.

**Read algorithm:**
1. `fs.statSync` the transcript. If `fileSize < cursor.fileSize` (truncate,
   compact) reset the cursor to `null`.
2. Parse all lines (malformed lines are skipped).
3. Locate `lastEmittedUuid` by UUID equality. Everything after it is new.
4. After successful post, save `{ lastEmittedUuid = last entry's uuid,
   byteOffset = fileSize, fileSize }`.

**Entry ‚Üí event mapping:**

| Transcript entry | Ingest kind | Lane | Notes |
|---|---|---|---|
| `assistant.message.content[].type === "thinking"` | `thought.logged` | (default) | body is the thinking text |
| `assistant.message.content[].type === "text"` when the same message also carries a `tool_use` block | `assistant.response` with `metadata.phase = "intermediate"` | (default) | intermediate narration; final-turn text comes from the primary `/api/assistant-response` post |
| `attachment.type === "task_reminder"` | `context.saved` | planning | `metadata.itemCount`, `metadata.content` |
| `attachment.type === "plan_mode"` | `context.saved` | planning | `metadata.planExists`, `planFilePath`, `isSubAgent`, `reminderType` |
| `attachment.type === "skill_listing"` | `instructions.loaded` | (default) | `metadata.skillCount`, body = listing text |
| `attachment.type === "deferred_tools_delta"` | `instructions.loaded` | (default) | `metadata.addedNames`, `removedNames` |
| `attachment.type === "mcp_instructions_delta"` | `instructions.loaded` | (default) | `metadata.addedNames`, `removedNames`, `addedBlocks` |
| `attachment.type === "nested_memory"` | `context.saved` | planning | `metadata.path`, `displayPath`, `memoryType`; body = memory content |
| `assistant.message.content[].type === "tool_use"` | skipped | ‚Äî | already posted by PostToolUse hooks |
| `user.message.content[].type === "tool_result"` | skipped | ‚Äî | implicit in hook success/failure |
| `attachment.type === "hook_success" / "async_hook_response" / "hook_additional_context" / "directory"` | skipped | ‚Äî | already represented by hook events |
| `system.subtype === "local_command"` | skipped | ‚Äî | out of scope for v1 |
| `permission-mode` | skipped | ‚Äî | out of scope for v1 |

**Deterministic messageId (idempotency):**
```
messageId = sha1("transcript:" + sessionId + ":" + assistantUuid + ":" + contentIndex).slice(0, 32)
```
Replaying the same transcript range produces the same IDs so duplicate posts
are harmless on the server side.

**Tool-use correlation (`metadata.toolUseId`):**
Every PostToolUse handler plus `PostToolUseFailure.ts` now reads `tool_use_id`
from the hook payload and threads it into event metadata as `toolUseId`. The
same id lives on the transcript's `tool_use.id` block ‚Äî the field that the
next-turn `thought.logged` event cites via `metadata.parentUuid` /
`assistantUuid` chaining ‚Äî so downstream consumers can rebuild the
`thought ‚Üí tool_use ‚Üí tool_result` chain without guessing.

## 15. OTLP token collection ‚Äî `adapter-otlp-logs`

Token usage is **not** read by the plugin hooks. Instead, Claude Code's built-in
OpenTelemetry exporter pushes per-API-call telemetry to the monitor's OTLP
endpoint, which converts it into `token.usage` timeline events.

### Activation (`.claude/settings.json`)

```json
{
  "env": {
    "CLAUDE_CODE_ENABLE_TELEMETRY": "1",
    "OTEL_LOGS_EXPORTER": "otlp",
    "OTEL_EXPORTER_OTLP_PROTOCOL": "http/json",
    "OTEL_EXPORTER_OTLP_LOGS_ENDPOINT": "http://127.0.0.1:3847/v1/logs",
    "OTEL_LOGS_EXPORT_INTERVAL": "5000"
  }
}
```

`CLAUDE_CODE_ENABLE_TELEMETRY=1` is the master switch ‚Äî all other OTEL vars are
ignored without it. Requires a Claude Code restart to take effect.

### Data flow


### Session ‚Üí task routing

Each OTLP log record carries a `session.id` attribute (the Claude Code session
UUID). This is the same value that `SessionStart.ts` registers via
`/api/runtime-session-ensure`, so it directly maps to a `taskId` through the
`runtime_session_bindings` table:

```
OTLP attrs["session.id"]  =  runtimeSessionId
  ‚îî‚îÄ‚ñ∫ runtime_session_bindings(runtimeSource="claude-plugin", runtimeSessionId)
        ‚îî‚îÄ‚ñ∫ taskId  +  monitorSessionId
```

**Late-arrival fallback**: `SessionEnd` clears `monitorSessionId` in the binding
before the OTLP batch window (5 s) expires. `resolveRuntimeBinding` handles this
by falling back to `findTaskId()` + most-recent session lookup when the primary
`find()` (which filters `monitorSessionId IS NOT NULL`) returns null.

### OTLP attributes extracted (`event.name === "api_request"`)

| Attribute | Field | Notes |
|---|---|---|
| `session.id` | `sessionId` | Required ‚Äî record skipped if absent |
| `input_tokens` | `inputTokens` | Defaults to 0 |
| `output_tokens` | `outputTokens` | Defaults to 0 |
| `cache_read_tokens` | `cacheReadTokens` | Defaults to 0 |
| `cache_creation_tokens` | `cacheCreateTokens` | Defaults to 0 |
| `cost_usd` | `costUsd` | Optional |
| `duration_ms` | `durationMs` | Optional |
| `model` | `model` | e.g. `"claude-sonnet-4-6"` |
| `prompt.id` | `promptId` | UUID identifying the API request; stored as metadata, not used for routing |

`session.id` falls back to the resource-level attribute if not present on the
individual log record.

## What the plugin never does

- Never writes to stdout (no `hookSpecificOutput`, no `additionalContext`)
- Never exits with code 2 (never blocks Claude Code)
- Never modifies tool input (`updatedInput`)
- All monitoring is purely observational ‚Äî Claude Code's behavior is unaffected
