# Agent Tracer Codex Adapter — Data Flow

This document describes how each Codex hook handler and the rollout observer process
data: what fields come from Codex on stdin (or the rollout file), what the adapter
generates itself, and what gets sent to the Agent Tracer monitor API.

---

## Architecture overview

```
Codex CLI                 Adapter (tsx)                   Monitor API
──────────────────────────────────────────────────────────────────────
  hook fires
      │
      ▼
  run-hook.sh ─────────── hooks/<handler>.ts
                               │
                               ├─ stdin: JSON payload (from Codex)
                               │
                               ├─ generates: titles, bodies, ids
                               │
                               └─ POST ─────────────────────────────► /api/* /ingest/v1/events


  SessionStart hook
      │
      └─ spawns (detached) ─── app-server/observe.ts
                                    │
                                    ├─ tails rollout JSONL
                                    │   (~/.codex/sessions/YYYY/MM/DD/rollout-*-<id>.jsonl)
                                    │
                                    └─ POST ─────────────────────────► /ingest/v1/events
                                         (context.snapshot on each token/rate-limit change)
```

Each hook process is short-lived. Every handler calls `/api/runtime-session-ensure`
(idempotent on the server) to resolve `taskId` and `sessionId` before posting events.

The rollout observer is a long-running background process spawned once per session
by `SessionStart`. It terminates on SIGINT or SIGTERM and is replaced if a stale
instance is detected on the next session start.

---

## Data origin legend

| Symbol | Meaning |
|--------|---------|
| **[CX]** | Value comes directly from Codex's hook payload (stdin) |
| **[GEN]** | Value is generated by the adapter at runtime |
| **[ENSURE]** | Value is returned from `/api/runtime-session-ensure` (re-called each hook; server is idempotent) |
| **[ENV]** | Value comes from an environment variable |
| **[FILE]** | Value is read from the rollout JSONL file |

---

## Semantic metadata

The adapter classifies each event using the shared inference library
(`shared/semantics/inference.*`). PostToolUse/Bash calls `inferCommandSemantic()` to
derive `lane`, `subtypeKey`, `subtypeLabel`, `subtypeGroup`, `toolFamily`, `operation`,
and `entityType`/`entityName`/`sourceTool` — all included in the ingest payload.

Two additional layers are applied before every `postTaggedEvent` call:
- `provenEvidence(reason)` stamps `evidenceLevel: "proven"` and `evidenceReason`
- `withTags(meta)` derives normalised `tags` from the full metadata object

## Session lifecycle

### 1. SessionStart → `hooks/SessionStart.ts`

**Codex provides [CX]:**
```
session_id        — unique session identifier
source            — "startup" | "resume"
model             — model in use (e.g. "o4-mini")
hook_event_name   — "SessionStart"
```

**Adapter generates [GEN]:**
```
title   — "Session started" (startup) | "Session resumed" (resume)
body    — "Codex CLI session started." | "Codex CLI session resumed."
trigger — same as source, stored as metadata
```

**Adapter writes to disk:**
```
<PROJECT_DIR>/.codex/agent-tracer/latest-session.json
  { sessionId, modelId?, source, updatedAt }
```

**Adapter spawns (detached background process):**
```
run-observer.sh --project-dir <PROJECT_DIR> --session-marker <session_id> --quiet
  → starts app-server/observe.ts to tail the rollout file
```

**API calls:**
```
POST /api/runtime-session-ensure
  runtimeSource     [GEN]  "codex-cli"
  runtimeSessionId  [CX]   session_id
  title             [GEN]  defaultTaskTitle() = "Codex CLI — <basename>"
  workspacePath     [ENV]  $CODEX_PROJECT_DIR

POST /ingest/v1/events  { kind: "context.saved", … }
  taskId    [ENSURE] from ensureRuntimeSession response
  sessionId [ENSURE] from ensureRuntimeSession response
  title     [GEN]   "Session started" | "Session resumed"
  body      [GEN]   description text
  lane      [GEN]   "planning"
  metadata:
    trigger [CX]   source ("startup" | "resume")
```

## Per-turn lifecycle

### 2. UserPromptSubmit → `hooks/UserPromptSubmit.ts`

**Codex provides [CX]:**
```
session_id   — unique session identifier
prompt       — raw text the user typed
model        — current model identifier (optional)
```

**Adapter generates [GEN]:**
```
messageId  — crypto.randomUUID()
title      — first 120 chars of prompt (ellipsized)
phase      — "initial" (taskCreated === true) | "follow_up"
captureMode — "raw"
source      — "codex-hooks"
```

**Adapter writes to disk:**
```
<PROJECT_DIR>/.codex/agent-tracer/latest-session.json
  { sessionId, modelId?, source: "user_prompt_submit", updatedAt }
```

**API calls:**
```
POST /api/runtime-session-ensure  (creates task on first message; resolves IDs)

POST /ingest/v1/events  { kind: "user.message", … }
  taskId    [ENSURE] from ensureRuntimeSession
  sessionId [ENSURE]
  title     [GEN]   truncated prompt (max 120 chars)
  body      [CX]    prompt (full text)
  lane      [GEN]   "user"
  metadata:
    messageId   [GEN]  UUID
    captureMode [GEN]  "raw"
    source      [GEN]  "codex-hooks"
    phase       [GEN]  "initial" | "follow_up"
```

---

### 3. Stop → `hooks/Stop.ts`

**Codex provides [CX]:**
```
session_id             — unique session identifier
last_assistant_message — final text of the model's response (optional)
model                  — model identifier used for the turn (optional)
```

> Unlike Claude Code, Codex does not expose `stop_reason` in the hook payload.
> The adapter uses a synthetic `"stop_hook"` value.

**Adapter generates [GEN]:**
```
messageId  — crypto.randomUUID()
title      — first 120 chars of last_assistant_message (or "Response (stop_hook)")
stopReason — "stop_hook" (synthetic; Codex does not expose stop_reason)
source     — "codex-cli"
```

**Adapter writes to disk:**
```
<PROJECT_DIR>/.codex/agent-tracer/latest-session.json
  { sessionId, modelId?, source: "stop", updatedAt }
```

**API calls:**
```
POST /api/runtime-session-ensure  (re-ensure to resolve taskId / sessionId)

POST /ingest/v1/events  { kind: "assistant.response", … }
  taskId    [ENSURE]
  sessionId [ENSURE]
  title     [GEN]   truncated response text
  body      [CX]    last_assistant_message  (omitted if absent)
  lane      [GEN]   "user"
  metadata:
    messageId  [GEN]  UUID
    source     [GEN]  "codex-cli"
    stopReason [GEN]  "stop_hook"

POST /api/runtime-session-end
  runtimeSource     [GEN]  "codex-cli"
  runtimeSessionId  [CX]   session_id
  summary           [GEN]  "Assistant turn completed (stop_hook)"
  completeTask      [GEN]  true
  completionReason  [GEN]  "assistant_turn_complete"
```

## Tool event lifecycle

### 4. PreToolUse → `hooks/PreToolUse.ts`

Fires before every Bash tool call. Its sole job is to guarantee that a task and
session exist in the monitor before PostToolUse/Bash tries to post against them,
avoiding a race condition.

**Codex provides [CX]:**
```
session_id  — unique session identifier
tool_name   — "Bash"
```

**API call:**
```
POST /api/runtime-session-ensure
  runtimeSource     [GEN]  "codex-cli"
  runtimeSessionId  [CX]   session_id
  title             [GEN]  defaultTaskTitle()
```

> This hook adds latency to every Bash command. The transport layer enforces a
> 2-second HTTP timeout on the ensure call.

---

### 5. PostToolUse: Bash → `hooks/PostToolUse/Bash.ts`

**Codex provides [CX]:**
```
session_id             — unique session identifier
tool_name              — "Bash"
tool_input.command     — the shell command that ran
tool_input.description — human-readable description (optional)
tool_use_id            — stable tool invocation ID (optional)
```

**Adapter generates [GEN]:**
```
title  — description || first 80 chars of command
body   — "description\n\n$ command" (or just command if no description)
lane   — inferred from command semantics
```

**API call:**
```
POST /ingest/v1/events  { kind: "terminal.command", … }
  taskId    [ENSURE]
  sessionId [ENSURE]
  lane      [GEN]   "exploration" | "implementation"
  title     [GEN]
  body      [GEN]
  metadata:
    evidenceLevel  [GEN]  "proven"
    evidenceReason [GEN]  "Observed directly by the Codex PostToolUse/Bash hook."
    subtypeKey     [GEN]  "shell_probe" | "run_test" | "run_lint" | "run_build" | "verify" | "run_command"
    subtypeLabel   [GEN]  human-readable label for subtypeKey
    subtypeGroup   [GEN]  "shell" | "execution"
    toolFamily     [GEN]  "terminal"
    operation      [GEN]  "probe" | "execute"
    entityType     [GEN]  "command"
    entityName     [GEN]  first token of the command string
    sourceTool     [GEN]  "Bash"
    tags           [GEN]  normalised tag array (via withTags)
    command        [CX]   tool_input.command
    description    [CX]   tool_input.description  (omitted if absent)
    toolUseId      [CX]   tool_use_id             (omitted if absent)
```

## Rollout observer

### 6. Background observer → `app-server/observe.ts`

The observer is a long-running Node process spawned by `SessionStart` via
`run-observer.sh`. It tails the Codex rollout JSONL file and forwards token-usage
and rate-limit snapshots to the monitor.

**Rollout JSONL path:**
```
~/.codex/sessions/<YYYY>/<MM>/<DD>/rollout-<timestamp>-<sessionId>.jsonl
```

**Rollout line types consumed:**
```
event_msg    { type: "event_msg",    payload: { type: "token_count", info, rate_limits } }
turn_context { type: "turn_context", payload: { turn_id, model } }
```

**Observer reads from disk [FILE]:**
```
<PROJECT_DIR>/.codex/agent-tracer/latest-session.json  — resolves session when no --session-marker
```

**Observer writes to disk:**
```
<PROJECT_DIR>/.codex/agent-tracer/observer.json
  { pid, sessionId, startedAt }   — written on startup (duplicate-guard)
  {}                               — overwritten on exit (SIGINT/SIGTERM/finally)
```

**Observer generates [GEN] per state change:**
```
Emits a contextSnapshot event when any of the following change:
  tokenUsage  — new cumulative and per-turn token counts (from event_msg / token_count)
  rateLimits  — new rate-limit window snapshots       (from event_msg / token_count)
  turnId      — new turn identifier                   (from turn_context)
  modelId     — new model identifier                  (from turn_context)
```

**API calls (on each state change):**
```
POST /api/runtime-session-ensure  (once on startup)

POST /ingest/v1/events  { kind: "context.snapshot", … }
  taskId    [ENSURE]
  sessionId [ENSURE]
  title     [GEN]   "Context <N>% used" | "Codex status snapshot"
  lane      [GEN]   "telemetry"
  metadata:
    source                        [GEN]   "codex-app-server"
    threadId                      [FILE]  Codex session id from rollout session_meta
    modelId                       [FILE]  model from turn_context
    modelProvider                 [FILE]  model_provider from rollout session_meta
    contextWindowUsedPct          [GEN]   last.totalTokens / modelContextWindow × 100
    contextWindowRemainingPct     [GEN]   100 − contextWindowUsedPct
    contextWindowTotalTokens      [FILE]  last_token_usage.total_tokens
    contextWindowSize             [FILE]  model_context_window
    contextWindowInputTokens      [FILE]  last_token_usage.input_tokens
    contextWindowOutputTokens     [FILE]  last_token_usage.output_tokens
    contextWindowCacheReadTokens  [FILE]  last_token_usage.cached_input_tokens
    reasoningOutputTokens         [FILE]  last_token_usage.reasoning_output_tokens
    lastTurnInputTokens           [FILE]  last_token_usage.input_tokens
    lastTurnOutputTokens          [FILE]  last_token_usage.output_tokens
    lastTurnCachedInputTokens     [FILE]  last_token_usage.cached_input_tokens
    lastTurnReasoningOutputTokens [FILE]  last_token_usage.reasoning_output_tokens
    rateLimitPrimaryUsedPct       [FILE]  rate_limits.primary.used_percent
    rateLimitPrimaryWindowDurationMins [FILE] rate_limits.primary.window_minutes
    rateLimitPrimaryResetsAt      [FILE]  rate_limits.primary.resets_at
    rateLimitSecondaryUsedPct     [FILE]  rate_limits.secondary.used_percent
    rateLimitSecondaryWindowDurationMins [FILE] rate_limits.secondary.window_minutes
    rateLimitSecondaryResetsAt    [FILE]  rate_limits.secondary.resets_at
    rateLimitFiveHourUsedPct      [GEN]   legacy alias when windowDurationMins === 300
    rateLimitSevenDayUsedPct      [GEN]   legacy alias when windowDurationMins === 10080
```

## Scope comparison vs Claude Code adapter

| Category | Claude Code adapter | Codex adapter |
|---|---|---|
| Session start | supported | supported |
| User prompt capture | supported | supported |
| Bash command capture | supported | supported |
| File tool capture | supported | not supported (no Codex hook) |
| MCP tool capture | supported | not supported (no Codex hook) |
| Web tool capture | supported | not supported (no Codex hook) |
| Token / context usage | supported | supported (via rollout observer) |
| Rate-limit snapshot | supported | supported (via rollout observer) |
| Instructions lifecycle | supported | not supported |
| Compact lifecycle | supported | not supported |
| Subagent lifecycle | supported | not supported |
| App-server notifications | N/A | ready (`normalize.ts`) — no live connection yet |

Current Codex hook coverage is intentionally narrower than Claude Code. The expansion
path for file/MCP/web capture is the Codex app-server JSON-RPC stream (`normalize.ts`),
not a wider hook surface.

## What the adapter never does

- Never writes to stdout (no `hookSpecificOutput`, no `additionalContext`)
- Never exits with code 2 (never blocks Codex)
- Never modifies tool input
- All monitoring is purely observational — Codex's behavior is unaffected
